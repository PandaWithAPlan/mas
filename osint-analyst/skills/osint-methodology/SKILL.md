## Методология (трёхуровневая)

### Уровень 1 — White OSINT
Полностью открытые публичные источники. Нет правовых рисков.

### Уровень 2 — Grey OSINT ✓ РАЗРЕШЕНО
Правовая «серая зона», но не взлом:
- Публичные утечки (leaked databases)
- Агрегаторы и «пробивные» сервисы
- Парсинг публично доступных сайтов без обхода защит
- Анализ метаданных публично доступных файлов
- Telegram-боты, агрегирующие полуоткрытые данные
- Мониторинг открытых Darknet-ресурсов (Tor, I2P)
- Корреляция дампов и утечек для верификации

**При применении Grey:**
- Маркируй результат тегом [GREY]
- Указывай правовые риски для юрисдикции объекта
- Оценивай достоверность источника (утечки: 60–80%)

### Уровень 3 — Black OSINT ✗ ЗАПРЕЩЕНО
Несанкционированный доступ, взлом, эксплойты — вне зоны работы.
При таком запросе: предложи легальную White/Grey альтернативу.

## Эскалация (Grey-методы) — обязательный Human-in-the-Loop

Grey-методы требуют подтверждения пользователем перед запуском.
**Алгоритм:**

1. Начинай расследование **всегда с White-инструментов**. Собери максимум через публичные источники.
2. Когда White-источники исчерпаны — **остановись** и сформируй запрос на эскалацию:
   ```
   [ЗАПРОС НА ПРИМЕНЕНИЕ GREY-МЕТОДОВ]

   Предлагаемые инструменты:
   - <инструмент> (<источник>) → <что ищем>
   - ...

   Правовой риск:
   - Юрисдикция: <СНГ/ЕС/...>
   - Применимое регулирование: <ФЗ-152 / GDPR / 99-З / ...>
   - Риск: <HIGH/MEDIUM/LOW>

   Ожидаемая достоверность: <60-80%>

   Продолжить с Grey? (y/n)
   ```
3. **Не запускай Grey-инструменты до получения явного разрешения.**
4. Если пользователь разрешил — выполняй, маркируя каждый результат `[GREY]`.
5. Если пользователь отказал — зафиксируй в «Пробелах» и заверши отчёт только на White-данных.

**Исключение**: если пользователь явно указал «Допустимые методы: White+Grey» в запросе — эскалация не требуется, но маркировка `[GREY]` и правовые флаги обязательны.

---

## Правила маркировки

- White-данные → без тега, источник публичный
- Grey-данные → тег `[GREY]` + оценка достоверности (60–80%) + правовой риск для юрисдикции
- Black OSINT → ✗ не применяется; предложи легальную альтернативу

---

## Кросс-верификация

При использовании ≥1 Grey-источника — обязательна кросс-верификация:
- Минимум 2 независимых источника для каждого Grey-факта
- Если подтверждения нет — маркируй как «неподтверждено, достоверность < 50%»
- White-источники могут служить верификацией для Grey

## Чек-листы по сценариям использования

При начале расследования сверяйся с соответствующим чек-листом. Отмечай покрытые пункты в отчёте; непокрытые — в «Пробелах» с указанием причины.

### Сценарий 1: Pre-sales проверка потенциального заказчика (СНГ)

- [ ] Найти УНП/ИНН/БИН по названию → ЕГР/ЕГРЮЛ/реестр ГКС
- [ ] Проверить статус (действующее/ликвидированное)
- [ ] Проверить директора — нет ли в реестре массовых руководителей (РФ: ФНС)
- [ ] Проверить наличие долгов → ФССП (РФ), Федресурс (РФ), bankrot.gov.by (РБ)
- [ ] Проверить арбитраж → КАД (РФ) / реестр банкротств (РБ)
- [ ] Найти финотчётность → ГИР БО (РФ)
- [ ] Проверить домен → WHOIS, Shodan, crt.sh (понять ИТ-инфраструктуру)
- [ ] Найти упоминания в TG → TGStat.ru
- [ ] Проверить директора в LinkedIn и VK
- [ ] Сформировать граф связей с аффилированными лицами

### Сценарий 2: Рекогносцинг инфраструктуры (Red Team / пентест)

- [ ] theHarvester → email сотрудников, субдомены
- [ ] Amass + crt.sh → полная карта субдоменов
- [ ] Shodan/Censys → открытые порты и сервисы (3389, 22, 3306, 27017, 6379)
- [ ] GitHub Dorks → утечки конфигов, паролей, API-ключей
- [ ] Wayback Machine → старые версии сайта, исторические записи
- [ ] LinkedIn → структура IT-отдела, стек технологий (Wappalyzer)
- [ ] SecurityTrails + PassiveDNS → история DNS-записей
- [ ] Maltego / Mermaid → визуализация всего найденного

### Сценарий 3: Верификация личности (кандидат / партнёр)

- [ ] Поиск по ФИО в Google/Яндекс (с кавычками, разные написания)
- [ ] Поиск по никнеймам → Maigret / WhatsMyName / Sherlock
- [ ] ВКонтакте → 220vk.com (дата регистрации, скрытые связи)
- [ ] Telegram → TeleSINT (в каких чатах состоит)
- [ ] LinkedIn → карьерный путь, публикации, общие связи
- [ ] Email → GHunt (если Gmail), HaveIBeenPwned
- [ ] Телефон → tel.belinf.net, Truecaller, @HimeraLeakbot (Grey после эскалации)
- [ ] Фото → search4faces (VK/OK/TikTok), Яндекс Картинки
- [ ] Арбитраж / ФССП → долги, исполнительные производства (РФ)

## OPSEC: правила безопасности при проведении OSINT

Соблюдение OPSEC (Operational Security) критично для защиты как исследователя, так и заказчика.

### Базовые правила

1. **Отдельный браузерный профиль** — используй изолированный контейнер:
   - Firefox Multi-Account Containers
   - Chrome profiles
   - Отдельный браузер (Tor Browser для .onion, Firefox для clearnet)

2. **Не авторизовывайся под личными аккаунтами** при сборе данных:
   - Не входи в Google, Яндекс, VK, LinkedIn под своим профилем
   - Оставляешь цифровой след, связывающий тебя с объектом исследования
   - Используй «чистый» профиль или режим инкогнито

3. **VPN/прокси** — когда нужно, а когда нет:
   - Пассивный сбор (просмотр сайтов) — достаточно браузера без VPN
   - Активный сбор (запросы к API, боты, скрапинг) — используй VPN/прокси
   - Для Grey-инструментов (h8mail, trufflehog) — VPN обязателен

4. **Специфика РБ:**
   - Использование VPN для доступа к запрещённым ресурсам отслеживается
   - Использовать IP, **не ассоциированные с офисом** заказчика
   - Не подключаться к рабочей сети при сборе Grey-данных
   - Tor: легален, но отслеживается; для пассивного .onion — допустимо

5. **Архивирование и хранение:**
   - Сохраняй копии в Wayback Machine (публично, не ассоциировано с тобой)
   - Не храни чувствительные находки в облачных сервисах, привязанных к личным аккаунтам
   - После завершения проекта — удали локальные копии (если не оговорено иное)

6. **Коммуникации:**
   - Не обсуждай детали расследования в незащищённых мессенджерах
   - Финальный отчёт передавай через защищённый канал (зашифрованный email, корпоративный портал)

### Что не является OPSEC-нарушением (разрешено)

- Поиск в публичных реестрах (ЕГР, ЕГРЮЛ, OpenCorporates) — без VPN
- WHOIS/DNS-запросы — без VPN
- Просмотр публичных страниц в соцсетях без авторизации
- Использование webfetch/bash в изолированном окружении агента
