Применяй на этапе Intake при приёме новой задачи.
Навык стандартизирует создание и ведение единого файла
`work-area/sessions/session-YYYY-MM-DD.md` —
главного хронологического документа задачи.

# Кто использует

| Агент | Роль | Действие |
|-------|------|----------|
| `team-lead` | Владелец | Создаёт и обновляет на протяжении всей задачи |
| `reviewer` | Читатель | Читает при подготовке финального отчёта (опционально) |

# Когда применять

## Team Lead

1. **При первом Intake задачи** — создать файл с нуля по шаблону ниже
2. **После оценки сложности** — зафиксировать балл и обоснование
3. **После выбора варианта пользователем** — зафиксировать вариант (A/B/C)
4. **При каждом REWORK-цикле** — добавить запись в «Журнал циклов»
5. **При инцидентах** (зависание агента, PARTIAL COMPLETION) — зафиксировать немедленно
6. **При ACCEPT / ESCALATE** — зафиксировать итоговый статус

Правило: запись в сессионный файл делается **сразу** после события,
а не в конце задачи. Файл — хроника, не ретроспектива.

## Reviewer

Читает файл если он существует. Если файла нет — не блокирует работу,
явно отмечает в отчёте: «История сессии недоступна».

# Шаблон файла

Путь: `work-area/sessions/session-YYYY-MM-DD.md`

```md
# Сессия — [краткое название задачи]

> Создана: YYYY-MM-DD HH:MM | Team Lead

## Метаданные задачи
- **Задача:** [краткое описание от пользователя]
- **Сложность:** [N/10] — [обоснование]
- **Выбранный вариант:** [A / B / C] — [одна строка: что выбрано] — ждёт выбора пользователя

## Хронология

| Время | Агент | Действие | Статус |

| HH:MM | explorer | Завершён — explore-report.md | DONE |
| HH:MM | analyst | Запущен | IN PROGRESS |
| HH:MM | analyst | Завершён — requirements.md | DONE |
| HH:MM | architect | Запущен — варианты A/B/C | IN PROGRESS |
| HH:MM | architect | Завершён — design.md | DONE |
| HH:MM | team-lead | Варианты переданы пользователю | WAITING |
| HH:MM | team-lead | Пользователь выбрал вариант N | — |
| HH:MM | devops | Preflight gate: окружение готово | DONE |
| HH:MM | coder-front | Запущен — DEV-FRONT-NNN | IN PROGRESS |
| HH:MM | coder-front | Завершён — статус DONE | DONE |
| HH:MM | coder-back | Запущен — DEV-BACK-NNN | IN PROGRESS |

| HH:MM | tester | Запущен — тест-кейсы | IN PROGRESS |
| HH:MM | tester | Завершён — TR-NNN.md | PASSED / FAILED / PARTIAL |
| HH:MM | guardian | Запущен | IN PROGRESS |
| HH:MM | guardian | Завершён — вердикт ACCEPT / REWORK | — |

| HH:MM | team-lead | Задача закрыта — ACCEPT | ✓ |

## Журнал циклов доработки

### Цикл 1 (основной)
- **Дата:** YYYY-MM-DD
- **Explorer:** explore-report.md — [кратко: что выявлено]
- **Analyst:** requirements.md — [кратко: ключевые требования]
- **Architect:** design.md — варианты A/B/C, рекомендован [X]
- **Тестирование:** TR-001.md — Passed: N, Failed: M, Skipped: K
- **Guardian:** вердикт ACCEPT / REWORK — [ключевые замечания]
- **Решение Team Lead:** ACCEPT / REWORK / ESCALATE

### Цикл 2 (доработка) — при наличии
- **Причина возврата:** [из Guardian / Tester]
- **Новые вводные:** [что изменилось в требованиях]
- **Explorer:** diff с циклом 1 — [что изменилось]
- **Analyst:** скорректированные требования — [что]
- **Architect:** скорректированный вариант [X] — [что изменено]
- **Тестирование:** TR-002.md — Passed: N, Failed: M
- **Guardian:** вердикт — [замечания]
- **Решение Team Lead:** ...

### Цикл 3 (финальная доработка) — при наличии
[та же структура]
- При вердикте REWORK после цикла 3 → ESCALATE пользователю

## Инциденты
- **HH:MM** — Агент: [имя] — Проблема: [описание] — Решение: [что сделано]
  - Пример: `tester` — завис на тест-кейсе TEST-003 — перезапущен с суженным scope

## Итоговый статус
- **Задача:** ACCEPTED / ESCALATED
- **Дата закрытия:** YYYY-MM-DD HH:MM
- **Финальный отчёт:** work-area/reports/final-report-NNN.md
```

# Правила заполнения

## Хронология
- Добавляй строку **перед** запуском агента (IN PROGRESS) и **после** его завершения (DONE / FAILED / PARTIAL)
- Агенты, работающие параллельно, отмечаются в одном временном слоте
- Если агент завершился с PARTIAL — пиши `PARTIAL (покрыто: X, не покрыто: Y)`

## Журнал циклов
- Каждый REWORK = новый раздел «Цикл N»
- Счётчик циклов начинается с 1 (основной), инкрементируется при каждом возврате
- Максимум 3 цикла, после чего обязательна ESCALATE

## Инциденты
- Фиксируй: зависания агентов, PARTIAL COMPLETION, неожиданные ошибки
- Каждый инцидент: временная метка, агент, проблема, решение
