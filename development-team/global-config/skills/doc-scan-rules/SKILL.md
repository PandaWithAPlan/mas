Применяй этот skill если в проекте есть папка `docs/`.

## Каноническое имя файла документации

ОДИН исходный файл = ОДИН файл документации. Имя выводится
детерминированно из пути к исходному файлу:

```
canonical_path = "docs/" + <путь к файлу от корня проекта,
                            "/" заменён на "-", расширение отброшено> + ".md"
```

Пример: `src/api/users/routes.py` → `docs/src-api-users-routes.md`.

Все агенты используют это же правило. Твоя задача — построить по нему
индекс (см. ниже), чтобы coder-агенты не угадывали имена и не плодили дубли.

## Что проверить

### 1. Модули без документации
Обойди все файлы в `src/`. Для каждого вычисли `canonical_path`
и проверь существование файла по этому пути.

Зафиксируй список:
- **Нет документации** — модуль существует, файл docs/ отсутствует
- **Документация устарела** — дата изменения файла docs/ старше
  даты последнего изменения модуля

Для каждого пропуска (нет документации / устарела) проставь приоритет —
это сигнал для Team Lead, заводить ли отдельную doc-задачу:
- **HIGH** — модуль с высокой связностью (на него ссылаются 3+ других
  модуля в диаграммах/импортах), или входит в критическую зону
  (auth, payment, API, данные пользователей, безопасность)
- **MED** — обычный модуль бизнес-логики со связями
- **LOW** — изолированный модуль, утилита, без входящих зависимостей

Приоритет вычисляется детерминированно по связности и зоне, не по вкусу.

### 2. Рассинхрон имён
Проверь что имена функций и классов в docs/ совпадают
с именами в актуальном коде. Фиксируй расхождения как `[OUTDATED]`.

### 3. Битые диаграммы
Найди Mermaid-блоки где ссылаются на несуществующие модули.

### 4. Дубли документации
Найди случаи, когда одному исходному файлу соответствует более одного
docs-файла (например, старый `docs/full-project-routes-note.md` и новый
`docs/src-api-users-routes.md` описывают один и тот же `routes.py`).
Это главный источник разрастания `docs/` — фиксируй явно.

## Формат вывода в explore-report.md

Добавь раздел в конце отчёта:

```md
## Состояние документации

### Индекс документации
| Исходный файл | canonical_path | Статус | Приоритет |
|---------------|----------------|--------|-----------|
| src/api/users/routes.py | docs/src-api-users-routes.md | актуально | — |
| src/core/db.py | docs/src-core-db.md | устарело | HIGH |
| src/models/user.py | docs/src-models-user.md | отсутствует | MED |

> Coder-агенты при обновлении документации берут `canonical_path`
> из этой таблицы — не вычисляют имя заново.
> Team Lead использует колонку «Приоритет», чтобы решить про doc-задачу.

### Модули без документации
- `src/path/module.ext` — документация отсутствует [приоритет: HIGH/MED/LOW]

### Документация устарела
- `docs/full-project-module-note.md` — код изменён [дата], docs [дата]

### Рассинхрон имён
- `docs/full-project-module-note.md` — функция `oldName` → в коде `newName`

### Битые ссылки в диаграммах
- `docs/full-project-module-note.md` → ссылается на несуществующий `ModuleX`

### Дубли документации
- `routes.py` документирован дважды: `docs/full-project-routes-note.md`
  и `docs/src-api-users-routes.md` → нужна консолидация
```

## Ограничения
- Не создавай и не изменяй файлы docs/ — только фиксируй проблемы
- Передай список проблем Team Lead как часть отчёта
- Team Lead решает включать ли обновление документации в текущую задачу
