> ⚙️ **Это навык-инструкция, а не функция.** Вызов `skill({ name: "complexity-parallel" })` один раз подгружает этот текст в твой контекст — у навыка нет отдельного исполнения, входных аргументов и возвращаемого значения. Прочитав процедуру, выполни её сам и встрой результат прямо в свой ответ или артефакт. Повторный вызов ради получения вывода — ошибка: ты снова получишь этот же текст.

Применяй на этапе Intake после оценки сложности задачи. Навык превращает
complexity score в конкретную конфигурацию параллелизации для Explorer,
Analyst, Architect и кодеров.

# Кто использует

| Агент | Когда |
|-------|-------|
| `team-lead` | После оценки сложности задачи (Intake → 1.1) |

# Концепция

Сложность задачи определяет не только объём работы, но и стратегию её выполнения.
Навык централизует правила:
- Как запускать Explorer (один/параллельно)
- Как запускать Analyst (один/разделение на функциональные/нефункциональные требования)
- Как запускать Architect (один с вариантами/параллельные варианты)
- Как декомпозировать задачи для кодеров
- Правила синхронизации после параллельных этапов

Кроме того, навык учитывает memory signals из global-memory.md —
накопленный опыт о том, какие стратегии параллелизации работали, а какие нет.

# Процедура

## Шаг 0. Собери входные данные

Эти данные у тебя уже есть в контексте (из задачи, артефактов и предыдущих шагов) — извне ничего не «передаётся». Собери их перед началом:
- `complexity_score` — оценка 0–10
- `task_zones` — функциональные зоны задачи (например, `["src/api/", "src/ui/"]`)
- `is_first_cycle` — первый цикл или повторный
- `memory_signals` — из global-memory.md, секции «Что сработало» / «Что не сработало»

## Шаг 1. Шкала сложности (справочно)

| Балл | Характеристика |
|------|----------------|
| 0–2  | Точечное изменение: один файл, изолированная правка, нет зависимостей |
| 3–4  | Локальная задача: 2–5 файлов, одна зона ответственности, минимальные зависимости |
| 5–6  | Средняя задача: несколько модулей, есть зависимости, требуется анализ влияния |
| 7–8  | Сложная задача: кросс-модульные изменения, нетривиальные зависимости, риски регрессии |
| 9–10 | Архитектурная задача: системные изменения, множество зон, высокий риск, требуется глубокий анализ |

## Шаг 2. Выбери конфигурацию Explorer

| Сложность | Режим запуска | Детали |
|-----------|---------------|--------|
| 0–4       | `single_full` | Один экземпляр, полный проход кодовой базы |
| 5–6       | `single_zoned` | Один экземпляр с явным указанием приоритетных зон для анализа |
| 7–10      | `parallel` | Параллельные экземпляры, каждый получает свой сегмент кодовой базы |

**Memory override:** Если `memory_signals.parallel_explorer_worked === true` —
используй `parallel` начиная со сложности 5+ (вместо 7+).

### Инструкции для parallel Explorer

1. Раздели кодовую базу на **непересекающиеся сегменты** по функциональным границам
   (например: `src/api/`, `src/ui/`, `src/core/`).
2. Каждому экземпляру передай:
   - Свой сегмент
   - Список точек интеграции с соседними сегментами
3. Укажи общий выходной файл: `work-area/docs/explore-report.md` —
   каждый агент дописывает свою секцию, не перезаписывая чужую.
4. После завершения всех экземпляров — прочитай отчёт целиком,
   убедись в отсутствии противоречий между секциями перед передачей Analyst.

## Шаг 3. Выбери конфигурацию Analyst

| Сложность | Режим запуска | Детали |
|-----------|---------------|--------|
| 0–6       | `single` | Один экземпляр анализирует все требования |
| 7–10      | `parallel_func_nonfunc` | Два экземпляра: функциональные требования + нефункциональные (производительность, безопасность, зависимости) |

**Инструкции для parallel Analyst:** Оба записывают в `work-area/docs/requirements.md`
в разные секции. Team Lead объединяет и проверяет отсутствие конфликтов перед передачей Architect.

## Шаг 4. Выбери конфигурацию Architect

| Сложность | Режим запуска | Детали |
|-----------|---------------|--------|
| 0–6       | `single` | Один экземпляр готовит 2–3 варианта решения |
| 7–10      | `parallel_variants` | Параллельные экземпляры: каждый прорабатывает один вариант решения независимо |

**Ограничение для повторных циклов:** Если `is_first_cycle === false` —
Architect корректирует **уже выбранный** вариант. Используй `single` независимо от
сложности (кроме случаев, когда выбранный вариант требует глубокой переработки
нескольких независимых компонентов — тогда допустим `parallel_variants` внутри одного варианта).

**Инструкции для parallel Architect:** Каждый агент пишет свой вариант в отдельную
секцию `work-area/docs/design.md`. Team Lead агрегирует и представляет пользователю
сравнительную таблицу.

## Шаг 5. Выбери конфигурацию кодеров

Кодеры (`coder-front` / `coder-back`) управляются независимо от шкалы сложности —
по принципу разделения ответственности (фронт/бэк).

При сложности 7–10:
1. Раздели задачу на **независимые подзадачи** внутри одной роли, если это возможно.
2. Явно зафиксируй в `work-area/tasks/`:
   - Границу каждой подзадачи
   - Список файлов, принадлежащих каждому исполнителю
3. **Пересечение файлов между параллельными исполнителями запрещено.**

**Memory override:** Если `memory_signals.parallel_coders_failed === true` —
не используй параллельных кодеров без явных файловых границ. При параллельном
запуске обязательно указывай непересекающиеся списки файлов.

## Шаг 6. Правила синхронизации (обязательны всегда)

После каждого параллельного этапа Team Lead обязан:
1. Прочитать все артефакты параллельных агентов.
2. Проверить отсутствие противоречий и дублирования между секциями.
3. Зафиксировать синтез в едином артефакте (или подтвердить его полноту).
4. Только после этого передавать задачу следующему агенту в цепочке.

## Шаг 7. Сформируй результат

Сформируй следующее **сам** и встрой в свой ответ/артефакт (это шаблон того, что ты пишешь, а не возвращаемое значение функции — повторно навык не вызывается):

Сформируй ответ в структуре:

```
{
  explorer: {
    mode: "single_full" | "single_zoned" | "parallel",
    segments: ["src/api/", "src/ui/", "src/core/"] | null,
    instructions: "..." | null
  },
  analyst: {
    mode: "single" | "parallel_func_nonfunc",
    segments: ["functional_requirements", "nonfunctional_requirements"] | null
  },
  architect: {
    mode: "single" | "parallel_variants",
    variant_count: 2 | 3,
    note: "повторный цикл — корректировка варианта X" | null
  },
  coders: {
    strategy: "standard" | "subtask_decomposition",
    subtasks: [{role: "front", files: [...]}, {role: "back", files: [...]}] | null,
    constraint: "no_file_overlap"
  },
  sync_rules: [
    "После параллельных Explorer: прочитать все секции, проверить противоречия",
    "После параллельных Analyst: объединить требования в единый документ",
    "После параллельных Architect: агрегировать варианты в сравнительную таблицу"
  ]
}
```

# Ограничения

- Не изменяй шкалу сложности — это прерогатива Team Lead на основе
  бизнес-контекста задачи
- При повторном цикле Architect не создаёт новый набор вариантов без
  явной необходимости
- Не игнорируй memory_signals — накопленный опыт важнее жёстких правил
