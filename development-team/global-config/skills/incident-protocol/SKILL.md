Применяй при любом инциденте в работе субагента: зависание, context limit,
невозможность завершить задачу в полном объёме.

# Кто использует

| Агент | Когда |
|-------|-------|
| `team-lead` | При зависании субагента или получении PARTIAL COMPLETION |

# Концепция

Инциденты в мультиагентной системе неизбежны: модель может упереться в context limit,
зависнуть на сложном тест-кейсе или не успеть завершить все разделы за отведённое время.
Протокол отвечает на три вопроса:
1. Достаточно ли того, что уже сделано, для передачи дальше?
2. Если недостаточно — как сузить scope для повторной попытки?
3. Когда нужно эскалировать пользователю?

# Процедура

## Шаг 1. Классифицируй инцидент

Определи тип:

- **subagent_hang** — агент не отвечает / циклится / превысил разумное время
- **partial_completion** — агент завершился, но с пометкой `⚠ PARTIAL COMPLETION`

## Шаг 2. Собери данные о покрытии

Из частичного артефакта или лога агента извлеки:

- `completed_sections` — что точно выполнено и зафиксировано
- `pending_sections` — что осталось невыполненным
- `failure_reason` — причина остановки (context limit, зависание на тесте, timeout)

## Шаг 3. Оцени достаточность

Используй таблицу решений:

| Доля покрытия | Тип агента | Решение |
|---------------|------------|---------|
| ≥80% | explorer, analyst, architect | SUFFICIENT — передай дальше с пометкой о частичном покрытии |
| ≥80% | tester | SUFFICIENT — передай Guardian с пометкой о непокрытых тест-кейсах |
| ≥80% | coder-front, coder-back | INSUFFICIENT — частичная реализация неприемлема, перезапуск |
| 50–79% | любой | INSUFFICIENT — перезапусти агента с суженным scope (только pending) |
| <50% | любой | INSUFFICIENT — перезапусти агента; если повторно <50% → ESCALATE |
| Критические секции не покрыты | tester, guardian | INSUFFICIENT — нельзя передавать непроверенные критические замечания |

### Критические секции (для tester/guardian)

Секция считается критической, если она затрагивает:
- **P0-тесты (для tester): покрытие P0 < 100%** — даже при общем покрытии ≥80% незакрытые P0 делают результат INSUFFICIENT (P0 = 100% это жёсткий KPI Tester'а)
- Безопасность (security issues из guardian-NNN.md)
- API-контракты (интерфейсные тесты)
- Зоны, помеченные как `affected_zones` в feedback.json

Если хотя бы одна критическая секция в `pending` — вердикт INSUFFICIENT независимо от доли покрытия.

## Шаг 4. Сформируй handoff note

### При SUFFICIENT

Сформируй явную пометку для следующего агента:

```
⚠ [Agent] завершён частично.
Покрыто: [completed_sections]
Не покрыто: [pending_sections]
Причина: [failure_reason]
Решение: продолжать с учётом непокрытых зон.
```

### При INSUFFICIENT

Сформируй суженный scope:

```
⚠ [Agent] не завершён. Перезапуск с суженным scope.
Новый scope: [pending_sections]
Причина: [failure_reason]
```

## Шаг 5. Выполни действие

### SUFFICIENT
- Передай handoff note следующему агенту в цепочке
- Зафиксируй инцидент в `work-area/sessions/session-YYYY-MM-DD.md` (секция «Инциденты»)

### INSUFFICIENT
- Запусти агента заново, передав только суженный scope
- Зафиксируй инцидент в сессии
- Если повторный запуск снова завершился PARTIAL:
  - При доле <50% — ESCALATE пользователю
  - Иначе — оцени повторно

### ESCALATE
- Не запускай агента повторно
- Подготовь для пользователя: имя агента, что выполнено, что нет, причина, рекомендация
- Жди решения пользователя

# Ограничения

- Никогда не передавай Guardian результаты без явной пометки о полноте тестирования
- Не увеличивай scope при перезапуске — только сужение
- Не скрывай инциденты — каждый фиксируется в сессии
