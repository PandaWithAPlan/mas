# Концепция

DevOps Gate — это точка обязательной проверки. До 4 разных агентов
дублировали чек-лист вручную, иногда с конфликтами правил (безусловный
vs условный вызов). Навык централизует логику: всем один алгоритм.

# Процедура

## Шаг 1. Определи сигналы для проверки

Оцени задачу по следующим признакам. Достаточно ответить ДА/НЕТ:

1. Требуется подготовка окружения? (новый сервис, миграция, конфигурация)
2. Затрагиваются БД, кэши, очереди или внешние сервисы?
3. Используются env-переменные? (новые или изменённые)
4. Нужны запуск, остановка или проверка доступности сервисов?

## Шаг 2. Прими решение

- Если **ни один** пункт не ДА — DevOps Gate не требуется.
- Если **хотя бы один** пункт ДА — DevOps Gate обязателен.

## Шаг 3. Выполни действие

### Gate не требуется

Зафиксируй в своём артефакте / сессии:

```
Devops Gate: не требуется (задача не затрагивает инфраструктуру)
```

Продолжай работу немедленно. **Не вызывай @devops.**

### Gate требуется

1. Вызови `@devops` с задачей:

   > Подготовь окружение для задачи [номер]. Проверь доступность сервисов,
   > подними что нужно, зафиксируй статус в devops-status.md.

2. Дождись явного ответа:
   - «Окружение готово» → продолжай работу
   - «Окружение не готово: [причина]» → сообщи Team Lead, не продолжай

## Шаг 4. Зафиксируй результат

В своём отчёте / сессии укажи:
- Результат проверки (gate required: да/нет)
- Какие критерии сработали (если required)
- Статус после вызова @devops (если вызывался)

# Ограничения

- Никогда не вызывай @devops если gate не требуется — это нарушает
  fallback-протокол и может заблокировать пайплайн
- Не продолжай работу без подтверждения если gate required
- Если @devops не отвечает 5 минут — сообщи Team Lead, не принимай
  решение самостоятельно
