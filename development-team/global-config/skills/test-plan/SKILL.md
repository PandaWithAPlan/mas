> ⚙️ **Это навык-инструкция, а не функция.** Вызов `skill({ name: "test-plan" })` один раз подгружает этот текст в твой контекст — у навыка нет отдельного исполнения, входных аргументов и возвращаемого значения. Прочитав процедуру, выполни её сам и встрой результат прямо в свой ответ или артефакт. Повторный вызов ради получения вывода — ошибка: ты снова получишь этот же текст.

Применяй в начале тестирования, после pre-flight проверок, до запуска первого теста.
Навык превращает плоский список тест-кейсов из TEST-NNN.md в приоритизированный план,
гарантируя, что критически важные тесты будут выполнены первыми.

# Кто использует

| Агент | Когда |
|-------|-------|
| `tester` | После `memory-retrieve`, `devops-gate-check` и чтения входных артефактов, перед началом выполнения тестов |

# Концепция

Тестирование в среде с ограниченным контекстом — это задача на управление риском:
не «успеть всё», а «успеть самое важное». Тест-план с приоритизацией гарантирует:
- Дымовые тесты (P0) — всегда первыми, их непокрытие = REWORK
- Регрессия в affected zones — приоритет выше, чем граничные случаи нового функционала
- При PARTIAL COMPLETION покрыто критическое ядро, остальное — осознанно отложено

# Процедура

## Шаг 0. Входные данные

Tester передаёт:

```
test_task_path: "work-area/tasks/TEST-NNN.md"
cycle_number: 1–3
feedback_memory: {                           // из feedback.json, опционально
  affected_zones: ["auth", "payment"],
  previous_failed_tests: ["TEST-003", "TEST-005"]
}
explore_zones_of_risk: [                     // из explore-report.md, секция «Зоны риска»
  "src/backend/auth/",
  "src/frontend/components/Checkout/"
]
```

## Шаг 1. Извлеки тест-кейсы из TEST-NNN.md

Прочитай файл `work-area/tasks/TEST-NNN.md` и выдели каждый тест-кейс.
Для каждого зафиксируй:
- `id` — идентификатор (например, TEST-001)
- `name` — название теста
- `description` — что проверяется
- `category` — функциональность / регрессия / граничный случай / интерфейсный контракт

Если test-task-файл ещё не создан Team Lead или отсутствует — верни ошибку:
«TEST-NNN.md не найден. Невозможно составить план тестирования. Сообщи Team Lead.»

## Шаг 2. Присвой приоритет каждому тест-кейсу

### Правила приоритизации

| Приоритет | Критерий | Пример |
|-----------|----------|--------|
| **P0** — дымовой/критический | Затрагивает affected zone из feedback.json | `TEST-003: login regression after auth change` в зоне `auth` |
| **P0** | Повторяет ранее проваленный тест из previous_failed_tests | `TEST-005` уже FAILED в цикле 1 |
| **P0** | Проверяет основной пользовательский сценарий (happy path) | `TEST-001: user can complete OAuth2 login flow` |
| **P0** | Проверяет интерфейсный контракт между фронтом и бэком | `TEST-007: frontend receives correct token format from /auth/login` |
| **P0** | Затрагивает security — аутентификация, авторизация, валидация ввода | `TEST-004: SQL injection in search field` |
| **P1** — основной | Проверяет альтернативный сценарий (второстепенный путь) | `TEST-008: login with expired token → redirect to re-auth` |
| **P1** | Проверяет регрессию в смежных, но не affected зонах | `TEST-010: user profile still works after auth changes` |
| **P1** | Проверяет обработку ошибок (graceful degradation) | `TEST-009: 500 on backend → frontend shows friendly error` |
| **P2** — граничный | Проверяет граничные значения | `TEST-012: email field with 255 characters` |
| **P2** | Проверяет регрессию в далёких зонах (low risk) | `TEST-015: homepage carousel still renders` |
| **P2** | Проверяет нестандартные конфигурации | `TEST-014: dark mode + RTL layout in login form` |

### Разрешение конфликтов приоритетов

Если тест подпадает под несколько правил — берётся **наивысший** приоритет.

### Для повторных циклов (cycle ≥ 2)

- Все тесты, которые были FAILED в предыдущем цикле → автоматически P0
- Новые тесты, добавленные в этом цикле → приоритет по общим правилам
- Тесты, которые были PASSED в предыдущем цикле → P1/P2 (в зависимости от категории)

## Шаг 3. Оцени сложность каждого теста

| Сложность | Характеристика | Маркер |
|-----------|----------------|--------|
| `light` | Проверить визуально / прочитать вывод / одна команда | ~1–2 минуты, минимум контекста |
| `medium` | Несколько шагов / сравнение данных / проверка состояния | ~3–5 минут |
| `heavy` | Многошаговый сценарий / требует анализа логов / несколько проверяемых условий | ~5–10 минут, расходует контекст |

## Шаг 4. Сформируй план выполнения

Упорядочи тест-кейсы:

1. **Все P0** — в порядке: security > интерфейсный контракт > happy path > affected zone регрессия > предыдущие провалы
2. **Все P1** — в порядке убывания сложности (лёгкие первыми — быстрый охват)
3. **Все P2** — последними, выполняются только при достаточном контексте

## Шаг 5. Выдай сводку и план

```markdown
## Тест-план — цикл [N]

### Сводка

| Приоритет | Количество | Сложность |
|-----------|-----------|-----------|
| P0 — критические | 4 | 1 light, 2 medium, 1 heavy |
| P1 — основные | 6 | 3 light, 2 medium, 1 heavy |
| P2 — граничные | 5 | 3 light, 2 medium |
| **Всего** | **15** | — |

### Рекомендуемая стратегия

1. **Выполни все P0** — это минимальный барьер для ACCEPT.
2. **Выполни P1** — основной охват функциональности.
3. **Если контекст позволяет** — выполни P2.
4. **При первых признаках истощения контекста** — завершай текущий приоритетный уровень и сохраняй PARTIAL COMPLETION.

### План выполнения

| # | ID | Название | P | Сложность | Обоснование |
|---|----|----------|---|-----------|-------------|
| 1 | TEST-007 | API contract: token format | P0 | light | Интерфейсный контракт фронт/бэк |
| 2 | TEST-004 | SQL injection in search | P0 | medium | Security |
| 3 | TEST-001 | OAuth2 login happy path | P0 | heavy | Основной сценарий |
| 4 | TEST-003 | Login regression in payment | P0 | medium | Affected zone (payment) |
| 5 | TEST-008 | Login with expired token | P1 | light | Альтернативный сценарий |
| 6 | TEST-010 | User profile after auth changes | P1 | light | Регрессия, смежная зона |
| 7 | TEST-009 | 500 → friendly error | P1 | medium | Обработка ошибок |
| ... | ... | ... | ... | ... | ... |
| 11 | TEST-012 | Email 255 chars | P2 | light | Граничный случай |
| ... | ... | ... | ... | ... | ... |
```

# Ограничения

- Не изменяй TEST-NNN.md — только читай
- Не пропускай тест-кейсы при формировании плана — все должны быть в плане
- Не меняй приоритеты произвольно — только по правилам
- При отсутствии feedback.json / explore-report — применяй только базовые правила (happy path → P0, граничные → P2)
