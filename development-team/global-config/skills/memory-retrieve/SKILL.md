> ⚙️ **Это навык-инструкция, а не функция.** Вызов `skill({ name: "memory-retrieve" })` один раз подгружает этот текст в твой контекст — у навыка нет отдельного исполнения, входных аргументов и возвращаемого значения. Прочитав процедуру, выполни её сам и встрой результат прямо в свой ответ или артефакт. Повторный вызов ради получения вывода — ошибка: ты снова получишь этот же текст.

Применяй в начале работы любого агента — сразу после получения задачи,
перед чтением других артефактов. Навык извлекает из памяти проекта
только то, что релевантно текущему агенту и текущей задаче,
не перегружая контекст.

# Кто использует

| Агент | Когда |
|-------|-------|
| Все агенты (`explorer`, `analyst`, `architect`, `coder-front`, `coder-back`, `tester`, `guardian`, `reviewer`) | Первым шагом при получении задачи |

# Концепция

MALMAS-паттерн: память — активный вход для агента, а не пассивный журнал.
Агент должен получить компактный, релевантный контекст, а не читать все файлы памяти целиком.

Навык решает проблему «что из накопленного опыта важно именно мне сейчас».

# Процедура

## Шаг 0. Определи свой Agent ID (MALMAS per-agent memory)

Ты работаешь в MALMAS-паттерне, где каждый агент имеет **собственную** память: ProcMem (procedural.json), FeedMem (feedback.json), ConMem (conceptual.md). Плюс общий GlobalMem (global-memory.md). Это значит, что ты должен фильтровать память по своему идентификатору агента.

Определи свой `agent_id` — ровно одно из:
- `explorer` | `analyst` | `architect` | `coder-front` | `coder-back`
- `tester` | `guardian` | `reviewer` | `devops` | `team-lead`

Все последующие шаги фильтруются по твоему `agent_id`:
- **procedural.json** → читай только записи с `"agent": "<твой_agent_id>"`
- **feedback.json** → читай записи, где `guardian.affected_zones` или `failed_details` относятся к зоне твоей ответственности (см. таблицу ниже)
- **conceptual.md** → читай секцию `### <твой_agent_id>`, затем общие эвристики без тега

Таблица соответствия agent_id ↔ зоны ответственности:

| agent_id | Зоны ответственности | Что искать в памяти |
|----------|---------------------|---------------------|
| `explorer` | Вся кодовая база | Зоны под риском, конфликтующие модули, рассинхрон документации |
| `analyst` | Требования, зависимости | Неясные зависимости, проблемные модули, открытые вопросы |
| `architect` | Архитектура, интерфейсы | Архитектурные ограничения, необратимые изменения, контракты |
| `coder-front` | `src/frontend/` | Запрещённые файлы, API-контракты с бэкендом, зоны регрессии фронтенда |
| `coder-back` | `src/backend/` | Запрещённые файлы, API-контракты с фронтендом, зоны регрессии бэкенда |
| `tester` | Тест-кейсы, качество | Affected zones, история провалов, граничные случаи |
| `guardian` | Security, correctness, код | Типовые уязвимости, категории проблем, affected zones |
| `reviewer` | Итоговые отчёты | Статус задачи, остаточные замечания, решение Team Lead |
| `devops` | Инфраструктура, окружение | Инфраструктурные решения, проблемы окружения, конфигурационные ограничения |
| `team-lead` | Вся задача целиком | Все слои, приоритет — global-memory.md |

## Шаг 1. Прочитать Global Memory

Файл `work-area/memory/global-memory.md` — обязателен всегда.
Это точка входа: статус задачи, ключевые ограничения, рекомендации.

Извлеки из него:
- Текущий статус задачи и номер цикла
- Ограничения (секция «Ограничения (на основе опыта)»)
- Рекомендацию «Для следующего агента» — особенно если она адресована тебе

## Шаг 2. Найти релевантные эвристики в Conceptual Memory (ConMem)

Если `work-area/memory/conceptual.md` существует:

1. **Per-agent секция:** прочитай секцию `### <твой_agent_id>` — это эвристики, специфичные для твоей роли (MALMAS-паттерн: каждый агент имеет свою ConMem)
2. **Общие эвристики:** прочитай эвристики без тега agent_id — они релевантны всем
3. **Учитывай ярус (tier, WI-3):**
   - `[active]` — полноценное знание; влияет на жёсткие решения/routing.
   - `[provisional]` — индуцирована из n=1, ещё не подтверждена baseline. Это **только
     мягкий хинт**: учитывай к сведению, но **не меняй из-за неё routing/scope** и не
     блокируй на ней решения. Жёсткий вес она получит лишь после промоута в `active`.
   - `[archived]` — **не применяй** (опровергнута или вытеснена; здесь по причине).
4. Найди эвристики по **совпадению ключевых слов** из твоей задачи:
   - Если задача про `auth` → ищи эвристики с `auth`, `login`, `OAuth`
   - Если задача про `payment` → ищи `payment`, `billing`, `transaction`
   - Если задача про `API` → ищи `endpoint`, `REST`, `input validation`

## Шаг 3. Проверить Feedback Memory на релевантные провалы (FeedMem)

Если `work-area/memory/feedback.json` существует:

1. Найди записи для текущей задачи (`task: "DEV-XXX"`)
2. **Per-agent фильтр:** ищи записи, где `guardian.affected_zones` пересекается с твоей зоной ответственности (см. таблицу в Шаге 0)
3. Из `failed_details` — какие тесты провалены и почему
4. Из `guardian.affected_zones` — какие зоны задеты критическими замечаниями

Пример: ты — `coder-back` и видишь, что `affected_zones: ["auth", "payment"]`,
а твоя задача — в `auth/`, ты должен знать, что `payment` под риском,
даже если явно не указан в задаче.

Если в feedback.json есть поле `"agent"` — читай только записи с `"agent": "<твой_agent_id>"`. Это MALMAS per-agent FeedMem.

## Шаг 4. Проверить Procedural Memory на дубликаты (ProcMem)

Если `work-area/memory/procedural.json` существует:

1. **Per-agent фильтр:** читай только записи с `"agent": "<твой_agent_id>"` (MALMAS: у каждого агента своя ProcMem). Это экономит контекст и исключает чужую историю.
2. Проверь: не делал ли ты уже то же действие с теми же файлами?
3. Если видишь дубликат (тот же файл, тот же change_type, статус FAILED) —
   не повторяй, сообщи Team Lead

## Шаг 5. Контекстный аудит (Meta-Contextualization)

На основе данных, собранных на шагах 1–4, проведи контекстный аудит — метаанализ
качества и применимости извлечённой информации. Цель: не просто прочитать память,
а оценить её релевантность, свежесть и согласованность с текущей задачей.

Сформируй для себя внутренний контекст в 5 подшагов (мысленно, без создания файлов):

### 5a. Анализ релевантности
- Какие эвристики из conceptual.md **прямо** связаны с текущей задачей? (ключевые слова, модули)
- Какие **косвенно** релевантны? (смежные модули, affected_zones из feedback.json)
- Что из памяти проекта явно **нерелевантно** этой задаче? (исключи из контекста)

### 5b. Проверка на устаревание
- Сравни источник эвристики (номер цикла) с текущим циклом:
  - Если эвристика из цикла N-3 или старше — проверь, не изменилась ли кодовая база с тех пор.
  - Если эвристика из раннего цикла, а структура модуля изменилась (см. explore-report diff) — пометь как **возможно устаревшую**.
- Если эвристика в conceptual.md не применялась 3+ циклов — явно отметь её как «низкоприоритетную».
- Если эвристика в ярусе `[archived]` (секция «Устаревшие / отозванные») — **не применяй**. Она там по причине.

### 5c. Выявление противоречий
- Есть ли эвристики, которые **противоречат** друг другу?
  - Пример: «AUTH-PAYMENT-COUPLING: любое изменение auth ломает payment» vs «PAYMENT-ISOLATED: payment больше не зависит от auth после рефакторинга в цикле 3».
  - При противоречии: доверяй более новой эвристике (больший номер цикла), но явно зафиксируй конфликт.
- Есть ли эвристики, противоречащие **текущему explore-report**?
  - Пример: эвристика говорит «модуль X нестабилен», а explore-report показывает, что X не менялся 5 циклов.

### 5d. Приоритизация контекстных сигналов
Отранжируй извлечённые сигналы по важности для твоей задачи:
```
[КРИТИЧЕСКИЙ] — active-эвристики (MANDATORY), BLOCKER-замечания Guardian, неистёкшие affected_zones
[ВАЖНЫЙ]     — active-эвристики с прямым совпадением ключевых слов, свежие (<2 циклов)
[ФОНОВЫЙ]    — provisional-эвристики (мягкие хинты), косвенно релевантные, старые (>3 циклов), общие рекомендации GlobalMem
```
> provisional никогда не поднимается выше [ФОНОВЫЙ] и не меняет routing/scope (WI-3).

### 5e. Фиксация пробелов
- Какая информация **отсутствует** в памяти, но была бы полезна для этой задачи?
- Зафиксируй мысленно: «После выполнения задачи стоит обновить память в части [X]».
- Это станет входом для `memory-update` → `memory-summarize` в конце цикла.

### Итоговый контекст (результат аудита)
```
[МОЙ КОНТЕКСТ ИЗ ПАМЯТИ — ПОСЛЕ АУДИТА]
- Задача: DEV-XXX, цикл N/3
- Ключевые ограничения: [2–3 пункта с пометкой приоритета]
- Релевантные эвристики: [1–3 с указанием свежести и возможных противоречий]
- Зоны под риском: [1–3 модуля]
- Что НЕ повторять: [дубликаты FAILED-действий]
- Устаревшие сигналы (игнорирую): [если есть]
- Противоречия (разрешены в пользу более новых): [если есть]
- Пробелы (что нужно обновить в памяти после задачи): [если есть]
```

# Ограничения

- Не читай все файлы памяти подряд, если они огромны — используй grep/search
  по ключевым словам из задачи
- Если какого-то файла памяти нет — пропусти соответствующий шаг, не блокируйся
- Ты не изменяешь файлы памяти — только читаешь
- Результат навыка — это твой внутренний контекст для принятия решений,
  а не отдельный артефакт
