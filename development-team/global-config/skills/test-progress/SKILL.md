Применяй после каждого выполненного тест-кейса (или группы из 3–5 тестов).
Навык принимает решение: продолжать выполнение, сузить scope, завершиться
с PARTIAL COMPLETION или нормально завершиться при полном покрытии (STOP_COMPLETE).

# Кто использует

| Агент | Когда |
|-------|-------|
| `tester` | После каждого тест-кейса / группы, перед тем как решить что делать дальше |

# Концепция

Главная причина PARTIAL COMPLETION у Tester'а — не «зависание», а истощение контекста.
Модель не всегда осознаёт, что контекст на исходе, и продолжает выполнять тесты,
пока не упрётся в limit и не оборвётся внезапно — с непредсказуемым покрытием.

`test-progress` решает эту проблему: после каждого теста он считает покрытие
по уровням приоритета и оценивает, сколько контекста ещё доступно.
Решение принимается осознанно, а не аварийно.

# Процедура

## Шаг 0. Входные данные

Tester передаёт после каждого тест-кейса:

```
test_plan: {                                  // из test-plan
  p0: {total: 4, completed: 2, remaining: 2},
  p1: {total: 6, completed: 1, remaining: 5},
  p2: {total: 5, completed: 0, remaining: 5}
}

last_test: {
  id: "TEST-003",
  status: "PASSED" | "FAILED" | "SKIPPED",
  complexity: "light" | "medium" | "heavy"
}

current_priority_level: "P0" | "P1" | "P2"

signals: {
  context_usage: "low" | "moderate" | "high" | "critical",
  response_latency_increasing: boolean,        // ответы модели стали медленнее?
  repetitive_actions_detected: boolean,        // rereading same files?
  tests_since_last_progress_check: 1–5        // сколько тестов после последнего check
}
```

### Как оценить context_usage

Tester оценивает субъективно, но по явным признакам:

| Уровень | Признаки |
|---------|----------|
| `low` | Контекст свободный, ответы быстрые, легко удерживаю все входные данные |
| `moderate` | Нормальный рабочий режим, всё читается без проблем |
| `high` | Начинаю забывать детали из начала сессии, перечитываю файлы, ответы дольше |
| `critical` | Еле удерживаю текущий тест-кейс, почти нет места для новых данных, модель повторяется |

## Шаг 1. Проверь критические условия немедленной остановки

Если хотя бы одно из условий истинно — **STOP_PARTIAL немедленно**, не продолжай:

- `context_usage === "critical"` — контекст на пределе
- `repetitive_actions_detected === true` — модель зациклилась
- Прошло >3 тестов с `context_usage === "high"` без улучшения

## Шаг 2. Оцени покрытие по приоритетам

### Минимальный барьер для CONTINUE

Для продолжения тестирования требуется:

| Условие | Порог |
|---------|-------|
| P0 покрытие | **100%** (все P0 должны быть выполнены) |
| P1 покрытие | ≥50% |
| P2 покрытие | любой (P2 — опциональны) |

Если P0 покрытие < 100% — решение зависит от текущего приоритетного уровня:
- Если сейчас выполняются **P0** — CONTINUE (доделать оставшиеся P0)
- Если сейчас выполняются **P1/P2**, а P0 ещё не все — это ошибка плана: STOP_PARTIAL и сообщи Team Lead

## Шаг 3. Прими решение

### Decision matrix

| P0 coverage | P1 coverage | context_usage | Решение |
|-------------|-------------|---------------|---------|
| 100% | ≥50% | low | **CONTINUE** — выполняй P1/P2 |
| 100% | ≥50% | moderate | **CONTINUE** — но следи за признаками |
| 100% | ≥50% | high | **SCOPE_REDUCE** → выполняй только оставшиеся P1, пропусти P2 |
| 100% | <50% | low / moderate | **CONTINUE** — добивай P1 |
| 100% | <50% | high | **SCOPE_REDUCE** → добей оставшиеся P1, затем STOP_PARTIAL |
| <100% | — | critical | **STOP_PARTIAL** — сохраняй что есть |
| 100% | 100% | любой | **STOP_COMPLETE** — все тесты плана выполнены (итог PASSED если все пройдены, иначе FAILED). НЕ PARTIAL |

### Действия по решению

#### CONTINUE

Продолжай выполнение тестов по плану. Следующий check через 3 теста
или при изменении context_usage.

#### SCOPE_REDUCE

- Пропусти все тесты с приоритетом P2
- Продолжай только оставшиеся P1 (и P0 если ещё есть)
- Зафиксируй в отчёте: «Scope сужен: P2 пропущены из-за высокого context_usage»
- Следующий check — после каждого теста

#### STOP_PARTIAL

1. Заверши текущий тест-кейс если он уже в процессе
2. Сформируй секцию PARTIAL COMPLETION в TR-NNN.md
3. Укажи явно:
   - Что покрыто (по уровням P0/P1/P2)
   - Что не покрыто
   - Причина остановки
4. Немедленно завершись

#### STOP_COMPLETE

Все тест-кейсы плана выполнены (P0+P1+P2 = 100%). Это нормальное завершение, **не** PARTIAL:

1. **Не** создавай секцию «⚠ PARTIAL COMPLETION» — покрытие полное, ничего не отложено
2. Проставь итог в TR-NNN.md: `PASSED` если все тесты прошли, иначе `FAILED`
3. Заверши тестирование

## Шаг 4. Верни решение

```
{
  decision: "CONTINUE" | "SCOPE_REDUCE" | "STOP_PARTIAL" | "STOP_COMPLETE",
  reasoning: "P0: 4/4, P1: 3/6, context_usage: moderate → CONTINUE: добить оставшиеся P1",
  coverage: {
    p0: {completed: 4, total: 4, percent: 100},
    p1: {completed: 3, total: 6, percent: 50},
    p2: {completed: 0, total: 5, percent: 0},
    overall: {completed: 7, total: 15, percent: 47}
  },
  next_check_after: 3,                        // через сколько тестов следующий check
  warning: null | "P0 не все выполнены — ошибка плана"
}
```

## Шаг 5. Формат PARTIAL COMPLETION секции

При STOP_PARTIAL добавь в TR-NNN.md:

```markdown
## ⚠ PARTIAL COMPLETION

### Выполнено
- P0: 4/4 (100%) — все критические тесты покрыты
- P1: 3/6 (50%) — TEST-008, TEST-010, TEST-009
- P2: 0/5 — не выполнялись

### Не выполнено
- TEST-011: password reset email format (P1)
- TEST-013: login via Google OAuth (P1)
- TEST-014: dark mode login form (P1)
- TEST-012: email 255 chars (P2)
- TEST-015: homepage carousel (P2)

### Причина остановки
Контекст приблизился к лимиту после выполнения 7 из 15 тестов.
План приоритизации выполнен: P0 покрыты полностью, P1 — на 50%.
Граничные случаи (P2) осознанно отложены.
```

# Ограничения

- Не продолжай тестирование при context_usage === "critical"
- Не принимай STOP_PARTIAL если P0 не все выполнены, а текущий уровень — P0
- Не пропускай проверку — минимальная частота: каждые 3 теста
- Не «оптимизируй» пороги — они основаны на эмпирике работы с context limit
