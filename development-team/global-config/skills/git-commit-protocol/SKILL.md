# Кто использует

| Агент | Роль в протоколе | Когда |
|-------|------------------|-------|
| `team-lead` | Оркестратор: формирует предложение, запрашивает согласие пользователя, делегирует исполнение | После ACCEPT + финального отчёта Reviewer |
| `devops` | Исполнитель: выполняет фактический `git commit` в режиме исполнителя | По явной задаче Team Lead, после согласия пользователя |

> **Почему два актора.** Team Lead владеет flow и общением с пользователем, но имеет
> `bash: deny` — сам git-команды выполнить не может. Фактическое исполнение `git commit`
> делает DevOps (`bash: allow`, режим исполнителя). Push не выполняет **никто** — см. Шаг 4.

# Концепция

`git commit` и `git push` затрагивают разделяемое состояние (репозиторий-хранилище кода
команды) и могут запустить CI/CD. Это действия с высокой ценой ошибки: коммит фиксирует
историю, push публикует её для всей команды и автоматики. До появления этого навыка момент
фиксации кода в VCS был не определён — flow завершался HTML-отчётом Reviewer, а кто и когда
коммитит код, оставалось на усмотрение, что небезопасно.

Протокол закрывает этот разрыв двумя жёсткими правилами:

1. **Commit — только с явного согласия пользователя и только в финале flow** (после ACCEPT
   и отчёта Reviewer). Промежуточные результаты, REWORK- и ESCALATE-циклы не коммитятся.
2. **Push — всегда ручное действие пользователя.** Ни один агент не делает `git push`. Агент
   готовит локальный коммит и подсказывает команду для push — выполняет push сам пользователь.

# Процедура

## Шаг 0. Проверь предусловие (gate)

Коммит разрешён, только если выполнены **все** условия одновременно:

1. Guardian вынес вердикт ACCEPT и Team Lead принял решение **ACCEPT**.
2. Reviewer сформировал финальный отчёт (`work-area/reports/final-report-NNN.md`).
3. Задача фактически завершена (критерии завершения достигнуты).

Если хотя бы одно условие не выполнено — **STOP, коммит запрещён**:
- REWORK / ESCALATE по умолчанию **не коммитятся**;
- исключение — пользователь **явно** просит закоммитить промежуточный результат или
  ESCALATE-результат с известными ограничениями. Только тогда продолжай, зафиксировав это
  как осознанное решение пользователя.

Зафиксируй в сессии: `Commit gate: passed (ACCEPT + Reviewer) | blocked (причина)`.

## Шаг 1. Сформируй предложение по коммиту (team-lead)

Не выполняя никаких git-команд, подготовь:

- **Список изменённых файлов** — из task-файлов (`DEV-*-NNN.md`) и отчётов кодеров: какие
  файлы реально менялись в этом цикле. Не из `git status` вслепую — только то, что входило
  в scope задачи.
- **Предлагаемое commit-сообщение** — краткое, по сути задачи (что и зачем сделано). Следуй
  стилю истории репозитория, если он известен.
- **Проверку на секреты** — убедись, что в список не попали секретные файлы (`.env`, ключи,
  токены, креды). Если такие файлы изменялись — исключи их из коммита и предупреди пользователя.

## Шаг 2. Запроси явное согласие пользователя (team-lead)

Покажи пользователю:
- список файлов, которые будут закоммичены;
- предлагаемое commit-сообщение;
- ветку, в которой будет сделан коммит.

Дождись **явного** «да». Молчание, неоднозначный ответ или общее «ок по задаче» ≠ согласие
на коммит. Пользователь вправе отредактировать сообщение или сузить список файлов — учти правки.

## Шаг 3. Делегируй исполнение коммита (team-lead → devops)

Поставь DevOps задачу в режиме исполнителя:

> Выполни локальный коммит для задачи [номер]. Файлы: [список]. Сообщение: «[текст]».
> Только `git add` указанных файлов и `git commit`. **БЕЗ `git push`.**

DevOps выполняет ровно это: `git add <файлы>` + `git commit -m "<сообщение>"`. Не добавляет
файлы вне списка, не делает push, не трогает другие ветки.

## Шаг 4. Push НЕ выполняется

После коммита **ни один агент не делает `git push`**. Team Lead сообщает пользователю:

> Изменения закоммичены локально в ветке `<branch>`. Push выполняется вами вручную:
> `git push origin <branch>`

Подсказку с командой дать можно — выполнять push нельзя. Это сознательная граница: решение
опубликовать код для команды и запустить CI/CD принимает человек.

# Ограничения

- Никогда не коммить **до** ACCEPT + финального отчёта Reviewer (кроме явного запроса пользователя).
- Никогда не коммить **без явного** согласия пользователя на сам коммит.
- **Никогда не делай `git push`** — это исключительно ручное действие пользователя, без исключений.
- Никогда не коммить секреты (`.env`, ключи, токены) — проверь staged-файлы до коммита.
- Team Lead **не выполняет git сам** (`bash: deny`) — только делегирует DevOps.
- DevOps коммитит **ровно** указанные Team Lead файлы — без «заодно» и без push.
- Не коммить файлы вне scope задачи, даже если `git status` показывает другие изменения.
