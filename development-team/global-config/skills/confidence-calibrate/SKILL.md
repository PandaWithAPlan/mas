> ⚙️ **Это навык-инструкция, а не функция.** Вызов `skill({ name: "confidence-calibrate" })` один раз подгружает этот текст в твой контекст — у навыка нет отдельного исполнения, входных аргументов и возвращаемого значения. Прочитав процедуру, выполни её сам и встрой результат прямо в свой ответ или артефакт. Повторный вызов ради получения вывода — ошибка: ты снова получишь этот же текст.

Применяй при формулировании значимых утверждений (findings, оценки рисков,
архитектурные рекомендации), где цена ошибки высока. Навык заменяет бинарные
и интуитивные суждения формальной метрикой уверенности 0–100%.

# Кто использует

| Агент | Когда |
|-------|-------|
| `guardian` | Для каждого finding в секции «Замечания» — перед присвоением severity |
| `architect` | Для оценок рисков, объёма изменений и обратимости в каждом варианте решения |

# Концепция

Бинарные выводы (CRITICAL / LOW RISK) скрывают неопределённость агента.
Guardian может пометить finding как CRITICAL при уверенности 40% —
Team Lead примет решение на основании сигнала, который слабо обоснован.

Навык калибрует каждое значимое утверждение по 4 критериям
(доступность данных, сложность, альтернативы, контекст) и выдаёт
взвешенную оценку, которую агент встраивает в свой артефакт.

# Процедура

## Шаг 1. Сформулируй утверждение (claim)

Опиши одним предложением: что именно ты утверждаешь?
- Guardian: «В файле X на строке Y есть SQL-инъекция»
- Architect: «Риск варианта B — высокий из-за зависимости от payment-модуля»

## Шаг 2. Собери evidence и gaps

Заполни две структуры:

**Evidence (что подтверждает):**
- Факты из кода/конфигов/артефактов, на которых основано утверждение
- Конкретные симптомы, строки, данные
- Подтверждения из результатов тестирования (если применимо)

**Gaps (что не проверено / неизвестно):**
- Какие файлы не были прочитаны
- Какие зависимости не проанализированы
- Какие допущения сделаны без проверки
- Ограничения контекста (context limit, недоступные файлы)

## Шаг 3. Оцени уверенность по 4 критериям

Для каждого критерия поставь оценку 0–25 баллов. Сумма = confidence 0–100%.

### Критерий 1. Доступность данных (0–25)
- 25: все релевантные файлы прочитаны, данные полны
- 15: основные файлы прочитаны, но есть пробелы в смежных модулях
- 5: данных мало, основано на косвенных признаках
- 0: утверждение основано на предположении без проверки

### Критерий 2. Сложность области (0–25)
- 25: область хорошо изучена, типовой паттерн, есть прецеденты в памяти
- 15: область знакома, но есть нюансы
- 5: незнакомая или редко встречающаяся область
- 0: область вне моей компетенции (например, инфраструктурный вопрос для coder-front)

### Критерий 3. Альтернативные интерпретации (0–25)
- 25: альтернативные объяснения проверены и исключены
- 15: одна альтернатива рассмотрена, но есть ещё незатронутые
- 5: альтернативы не рассматривались, вывод может быть ошибочным
- 0: есть как минимум одна правдоподобная альтернатива, которую я не могу исключить

### Критерий 4. Ограничения контекста (0–25)
- 25: контекст полный, все артефакты прочитаны, context limit не достигнут
- 15: контекст достаточный, но некоторые артефакты прочитаны частично
- 5: контекст сильно ограничен (PARTIAL COMPLETION, обрезанные файлы)
- 0: контекст критически неполон, утверждение ненадёжно

## Шаг 4. Сформируй калиброванную оценку

Вычисли `confidence = К1 + К2 + К3 + К4` (0–100).

Сформулируй одно из:

```
confidence ≥ 80%: «высокая уверенность» — утверждение надёжно, можно принимать решения
confidence 60–79%: «средняя уверенность» — утверждение вероятно верно, но требует оговорок
confidence 40–59%: «низкая уверенность» — утверждение под вопросом, требуется дорасследование
confidence < 40%: «недостоверно» — утверждение не должно быть основанием для решений без проверки
```

## Шаг 5. Встрой в артефакт

### Guardian: в каждое finding добавить confidence

```
### SIGNIFICANT [confidence: 70% — средняя]
- `src/backend/auth/login.py:42` — отсутствует rate limiting на брутфорс
  → добавить middleware с ограничением 5 попыток/минуту
  - Evidence: прочитан файл login.py, виден вызов authenticate() без счётчика попыток
  - Gaps: не проверен nginx.conf (возможно, rate limiting на уровне reverse proxy)
  - Calibration: данные 15 + сложность 20 + альтернативы 20 + контекст 15 = 70%
```

### Architect: в оценки вариантов добавить confidence

```
| Критерий | Вариант A | Вариант B | Вариант C |
|----------|-----------|-----------|-----------|
| Риски | Средний (conf. 65%) | Высокий (conf. 55%) | Низкий (conf. 80%) |
| Обратимость | Высокая (conf. 90%) | Средняя (conf. 60%) | Низкая (conf. 75%) |
```

# Ограничения

- Не калибруй рутинные утверждения — только значимые (findings с severity ≥ SIGNIFICANT, оценки рисков, ключевые рекомендации)
- Не подменяй confidence формальным severity: severity = критичность последствий, confidence = уверенность в наличии проблемы
- BLOCKER с confidence 20% — это сигнал Team Lead: «возможно, критично, но нужно перепроверить»
- MINOR с confidence 95% — надёжное замечание, но не блокирующее
- Если evidence пуст (только догадка) — confidence автоматически < 40%, не пытайся натянуть

# Self-check

- [ ] Claim сформулирован одним предложением
- [ ] Evidence содержит конкретные факты (файл, строка, симптом)
- [ ] Gaps честно перечисляют непроверенное
- [ ] 4 критерия оценены, confidence посчитан
- [ ] Confidence встроен в артефакт с кратким обоснованием
- [ ] При confidence < 40% — явно указано «недостоверно, требует проверки»
