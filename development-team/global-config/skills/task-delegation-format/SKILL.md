> ⚙️ **Это навык-инструкция, а не функция.** Вызов `skill({ name: "task-delegation-format" })` один раз подгружает этот текст в твой контекст — у навыка нет отдельного исполнения, входных аргументов и возвращаемого значения. Не жди, что навык «вернёт» `delegation_message`: прочитав процедуру, сам сформируй сообщение делегации и встрой его в свой ответ. Повторный вызов ради получения вывода — ошибка: ты снова получишь этот же текст.

Применяй каждый раз, когда Team Lead ставит задачу другому агенту.
Навык обеспечивает единый стандарт делегирования: структура задачи,
контроль передаваемого контекста и соблюдение границ владения артефактами.

# Кто использует

| Агент | Когда |
|-------|-------|
| `team-lead` | При каждой постановке задачи подчинённому агенту |

# Концепция

Team Lead — не исполнитель, а координатор. Качество его делегирования определяет
качество всей цепочки. Три проблемы, которые решает навык:
1. **Неполная постановка** — агент не получает всех 7 обязательных элементов
2. **Context bloat** — в сообщение вставляются полные файлы вместо ссылок
3. **Нарушение владения** — агенты модифицируют чужие артефакты

# Процедура

## Шаг 1. Собери входные данные

Эти данные у тебя уже есть в контексте (из задачи пользователя, сессии, памяти проекта и design.md) — извне ничего не «передаётся». Собери их перед формированием делегации:

```
target_agent: "explorer" | "analyst" | "architect" | "coder-front" | "coder-back" | "tester" | "guardian" | "reviewer" | "devops"

goal: "одно предложение — что нужно сделать"

context: "почему, в каком цикле, какие предшествующие решения"

constraints: [
  "не трогать X",
  "соблюдать контракт Y",
  ...
]

expected_artifact: "work-area/docs/explore-report.md"

completion_criteria: [
  "пункт 1",
  "пункт 2",
  ...
]

read_paths: [
  "work-area/docs/design.md",
  "work-area/memory/global-memory.md",
  ...
]

write_path: "work-area/docs/explore-report.md"

cycle_number: 1 | 2 | 3 | null

is_rework: boolean

rework_reason: "TEST-003 regression in payment" | null
```

## Шаг 2. Inject Memory (MALMAS Prompt Construction)

MALMAS-паттерн (Dong et al., 2026, arXiv:2604.20261, Algorithm 1): prompt агента =
`metadata + FeedMem + ConMem + GlobalMem`.

Перед формированием delegation message извлеки из памяти проекта контекст,
релевантный целевому агенту:

1. **GlobalMem:** прочитай `global-memory.md` → выбери пункты из секций
   «Ограничения» и «Для следующего агента», адресованные target_agent
2. **ConMem:** если `conceptual.md` существует → извлеки эвристики
   из секции `### <target_agent>` или общие cross-agent эвристики
3. **FeedMem:** если `feedback.json` существует → извлеки affected_zones
   и проваленные тесты, релевантные зоне ответственности target_agent

Сформируй блок «Контекст из памяти» и вставь его в delegation message
между «Контекст» и «Ограничения»:

```markdown
**Контекст из памяти (MALMAS):**
- **Что сработало:** [из GlobalMem → «Что сработало», если адресовано агенту]
- **Эвристики для тебя:** [из conceptual.md → секция target_agent]
- **Зоны под риском:** [из feedback.json → affected_zones]
- **Что НЕ повторять:** [из conceptual.md → эвристики с пометкой FAILED или из procedural.json → дубликаты]
```

Если память пуста (первый цикл, нет файлов) — замени блок на:
```markdown
**Контекст из памяти (MALMAS):** память проекта пуста (первый цикл).
```

## Шаг 3. Сформируй delegation message

Всегда включай 7 обязательных элементов:

```markdown
## Задача для [target_agent]

**Цель:** [goal]

**Контекст:** [context]
[если is_rework: Цикл доработки N/3 — причины возврата: rework_reason]

**Ограничения:**
- [constraint 1]
- [constraint 2]

**Прочитать:**
- [read_path 1]
- [read_path 2]

**Сформировать:** [expected_artifact] → сохранить в [write_path]

**Критерий завершения:**
- [completion_criterion 1]
- [completion_criterion 2]
```

### Особые случаи по типам агентов

| target_agent | Что добавить |
|-------------|-------------|
| `explorer` | При повторном цикле: «Прочитай предыдущий explore-report.md и зафиксируй diff» |
| `analyst` | «Основа — explore-report.md. Не домысливай структуру проекта» |
| `architect` | «Строго на базе существующего кода из explore-report.md. Ты НЕ создаёшь task-файлы» |
| `coder-front` / `coder-back` | «Change budget: [N] файлов из списка в design.md. Файлы вне списка — не трогать. Если нужно больше — сообщи Team Lead. Файлы, похожие, но НЕ трогать: [из design.md]» |
| `tester` | «Перед тестами вызови skill(devops-gate-check). Только функциональность, не качество кода» |
| `guardian` | «Запускается строго после Tester. TR-NNN.md обязателен» |
| `reviewer` | «Пиши для пользователя. Не упоминай агентов и циклы. Session-файл опционален» |
| `devops` | «Режим: исполнитель. Файлы: инфраструктурные. Секреты не раскрывать» |

## Шаг 3.5. PSAO Gate — предотвращение семантического дрейфа

PSAO-аннотирование оборачивает delegation message в структуру с мягкими директивами
(`[критически важно]`, `[контекст: ...]`, `[высокий приоритет]`),
которые направляют внимание агента-получателя и предотвращают потерю ключевых
ограничений при чтении длинного промпта.

### 3.5.1. Оценка необходимости PSAO

PSAO применяется, если выполняется ХОТЯ БЫ ОДНО из условий:

| Условие | Причина |
|---------|---------|
| `complexity ≥ 5` | Сложные задачи имеют много ограничений — без PSAO часть потеряется |
| `is_rework = true` | Повторный цикл: у агента уже есть контекст, и новые вводные могут «утонуть» |
| Наличие «критических» ограничений | Если в constraints есть слова: «критично», «обязательно», «запрещено», «только», «строго», «ни в коем случае» |
| `target_agent` = `coder-front` или `coder-back` с change budget | SCOPE GATE: потеря файловых границ = scope creep |
| `target_agent` = `architect` | Проектировщик получает много вводных, аннотации предотвращают дрейф от требований Analyst |

### 3.5.2. Применение PSAO

Если Gate принял решение ПРИМЕНИТЬ:

1. Подгрузи навык `skill({ name: "psao-annotate" })` (один раз) — он содержит процедуру аннотирования. Навык ничего не возвращает: ты применяешь его процедуру **сам** к данным, которые у тебя уже есть:
   - `original_prompt`: полный текст delegation message из Шага 3
   - `intent`: строка goal из delegation
   - `critical_constraints`: извлеки из constraints сообщения строки, содержащие маркеры критичности
   - `complexity`: оценка сложности
   - `is_rework`: true/false

2. По процедуре навыка **сам сформируй** `psao_prompt` в формате `plain` (без XML-тегов, только суффиксные аннотации `[]`). Не вызывай навык повторно ради «возврата» — это вернёт тот же текст инструкции.

3. **Замени** delegation message на `psao_prompt`. Убедись, что:
   - 7 обязательных элементов сохранены (цель, контекст, ограничения, прочитать, сформировать, критерии)
   - Критические ограничения помечены `[критически важно]`
   - Change budget помечен `[строго: максимум N файлов]`
   - Цель помечена `[очень важно]`

### 3.5.3. Пропуск PSAO

Если ни одно условие Gate не выполнено (простая задача, complexity 1-3, нет критических
ограничений, первый цикл, не кодер с change budget) — **пропусти PSAO** без вызова навыка.
Зафиксируй: `psao_gate: SKIPPED — complexity=2, is_rework=false, no critical constraints`.

## Шаг 4. Примени контроль контекста

При формировании сообщения соблюдай правила:

1. **Не вставляй полное содержимое больших файлов** в сообщение.
   Агент может прочитать их сам по указанным путям.

2. **Передавай пути к файлам и краткую выжимку**, а не полный текст.

3. **Для повторного цикла** — только новые вводные, причины возврата
   и ссылки на релевантные артефакты. Не дублируй весь контекст первого цикла.

4. **Если артефакт уже существует** — агент должен обновлять его,
   а не пересоздавать рассуждение с нуля. Укажи это явно: «Обнови существующий файл,
   не пересоздавай».

## Шаг 5. Проверь владение артефактами

Убедись, что делегируемая задача не нарушает границы владения:

| Артефакт | Владелец | Кому можно читать | Кому можно изменять |
|----------|----------|-------------------|---------------------|
| `work-area/tasks/DEV-*.md` | `team-lead` | Все агенты | **Только team-lead** |
| `work-area/tasks/TEST-*.md` | `team-lead` | Все агенты | **Только team-lead** |
| `work-area/docs/explore-report.md` | `explorer` | Все агенты | `explorer` (team-lead по команде) |
| `work-area/docs/requirements.md` | `analyst` | Все агенты | `analyst` |
| `work-area/docs/design.md` | `architect` | Все агенты | `architect` |
| `work-area/reports/test-results/TR-NNN.md` | `tester` | Все агенты | `tester` |
| `work-area/reports/guardian-NNN.md` | `guardian` | Все агенты | `guardian` |
| `work-area/reports/final-report-NNN.md` | `reviewer` | Все агенты | `reviewer` |
| `work-area/memory/*` | `team-lead` | Все агенты | **Только team-lead** |

Если задача требует, чтобы агент изменил **чужой** артефакт — такое допустимо только
по явной команде Team Lead. В delegation message добавь: «Разрешено изменить [артефакт]
по указанию Team Lead».

Если агент — **не владелец** task-файла, но в constraints указано взаимодействие
с task-файлом, укажи режим: «Только чтение».

## Шаг 6. Сформируй результат

Сформируй следующее **сам** и встрой в свой ответ (это шаблон того, что ты пишешь, а не возвращаемое значение функции — повторно навык не вызывается):

```
{
  delegation_message: "## Задача для [target_agent]\n\n**Цель:** ...",
  psao_applied: true | false,
  psao_annotations_count: N | null,
  context_rules_applied: [
    "файлы переданы как пути, не как содержимое",
    "только новые вводные для повторного цикла",
    ...
  ],
  artifact_ownership_checked: true,
  warnings: [
    "coder-back запрашивает изменение explore-report.md — требуется явное разрешение Team Lead"
  ] | []
}
```

# Ограничения

- Не делегируй задачу агенту без всех 7 элементов
- Не нарушай владение артефактами без явной команды Team Lead
- Не вставляй полные файлы в delegation message
