# Мультиагентная система разработки для opencode

Иерархическая мультиагентная система (orchestrator-worker) для [opencode](https://opencode.ai), построенная по принципам **loop engineering** с человеко-гейтнутым, аудируемым циклом. Один координирующий агент (`team-lead`) принимает задачу, декомпозирует её и проводит через детерминированный конвейер специализированных агентов — от исследования кодовой базы до финального ревью — с внешней верификацией, ограниченным циклом доработки и устойчивой памятью вне контекста.

Конфигурация разделена на два слоя:

- **глобальный** (`global-config/`) — переносимая, безопасная по умолчанию основа: роли агентов, скиллы, общие правила и жёсткая least-privilege модель прав. Не зависит от конкретного проекта.
- **проектный** (`project-config/`) — то, что адаптируется под каждый репозиторий: пути исходников для кодеров, LSP-серверы и контекст проекта.

> Эта схема рассчитана на то, что глобальный слой ставится один раз, а проектный — копируется и правится под каждый новый проект. Раздел [«Адаптация под свой проект»](#адаптация-под-свой-проект) — основной для тех, кто внедряет систему у себя.

## Архитектура

### Состав агентов

| Агент | Роль | Модель (пример) |
|---|---|---|
| `team-lead` | Координатор. Единственная точка входа. Декомпозиция, маршрутизация, решение ACCEPT/REWORK/ESCALATE, запись памяти. Сам код не пишет и не исполняет. | `nemotron-3-ultra` |
| `explorer` | Картирование кодовой базы, граф зависимостей. Только чтение. | `deepseek-v4-flash` |
| `analyst` | Извлечение требований. | `qwen3.5:397b` |
| `architect` | Проектирование решения, выбор варианта. | `deepseek-v4-pro` |
| `coder-front` | Реализация фронтенда. | `kimi-k2.7-code` |
| `coder-back` | Реализация бэкенда. | `kimi-k2.7-code` |
| `tester` | Прогон тестов, отчёт о результатах. | `glm-5.1` |
| `guardian` | Аудит качества и безопасности, вердикт. | `minimax-m3` |
| `devops` | Префлайт-проверка окружения, инфраструктура. | `gemini-3-flash-preview` |
| `reviewer` | Финальный отчёт для пользователя. | `minimax-m2.7` |
| `ai-engineer` | Отладка и RCA. Вне конвейера, вызывается вручную. | `deepseek-v4-pro` |

### Конвейер

```
Explorer → Analyst → Architect → [DevOps: префлайт-гейт] → Coder(s) → Tester → Guardian → [Team Lead: решение] → Reviewer
```

`ai-engineer` стоит отдельно от конвейера и применяется для разбора инцидентов.

### Верификация

Успех определяется **внешней валидацией**, а не самооценкой исполнителя. `tester` прогоняет реальные тесты и фиксирует результат, `guardian` проводит независимый аудит поверх кода и тестов и выносит рекомендательный вердикт с градацией серьёзности. Окончательное решение принимает `team-lead`:

- **BLOCKER** — непереопределяем, доработка обязательна;
- **CRITICAL** — переопределяем только с документированным обоснованием;
- ниже — на усмотрение координатора.

Помимо инженерных severity, действует **функциональный гейт**: провал любого P0-теста блокирует ACCEPT наравне с BLOCKER (непереопределяемо), а общий итог тестирования FAILED не может быть принят без устранения. Гейт встроен в `guardian-verdict` (Правило 0) и продублирован в предусловии ACCEPT у `team-lead` — чтобы известный падающий тест не прошёл в приёмку даже при пустом списке замечаний Guardian.

### Завершаемость цикла

Цикл доработки огорожен с трёх сторон, чтобы исключить бесконечное вращение и неконтролируемый расход токенов:

- жёсткий лимит циклов REWORK;
- детекция плато (стагнация результата имеет приоритет над счётчиком циклов);
- инцидент-протокол с порогами по покрытию и эскалацией к пользователю.

### Память

Трёхслойная модель памяти живёт **в файлах вне контекста** (а не «подстроками» в окне):

- **процедурная** — что было сделано, кем, с какими файлами;
- **обратной связи** — выводы из доработок, затронутые зоны;
- **концептуальная** — устойчивые знания о проекте.

Первое действие каждого агента — извлечение релевантной памяти; запись и сведение памяти — ответственность `team-lead`.

### Навыки (skills)

Поведение агентов вынесено в переиспользуемые **навыки** (`global-config/skills/*/SKILL.md`) — отдельные процедуры под конкретные паттерны работы (делегирование, аннотирование промптов, severity-скоринг, сведение памяти и т.д.). Дизайн сознательно модульный: новая техника промптинга или паттерн добавляется **отдельным маленьким навыком**, что упрощает масштабирование и быстрый откат при неудаче.

Ключевой момент модели исполнения: **навык — это инструкция, подгружаемая в контекст, а не вызываемая функция.** Вызов `skill({ name: "X" })` один раз вставляет текст `SKILL.md` в контекст вызывающего агента; у навыка нет отдельного исполнения, аргументов и возвращаемого значения — агент сам выполняет процедуру и встраивает результат в свой ответ. Это правило зафиксировано в `AGENTS.md` (секция «How Skills Work»), и каждый `SKILL.md` открывается шапкой-предохранителем об этом.

Сводная **карта навыков** — `global-config/skills/README.md`: реестр всех навыков (назначение, кто вызывает, триггер, порядок, фолбэк) как единый источник истины по диспетчерскому слою. Целостность «каждый навык ↔ строка в карте» проверяет `skills/check-skill-map.sh`.

## Модель безопасности

Права реализованы по принципу наименьших привилегий и держатся на движке opencode, а не только на инструкциях в промптах.

**Глобальные предохранители (для всех агентов):**

- запрет на чтение секретов: `**/.env`, `**/.env.*`, `**/*.key`, `**/*.pem`, `**/id_rsa*`, `**/*.secret`, `**/secrets/**`, `**/.aws/**`, `**/.ssh/**`;
- `external_directory: deny` — запрет на работу вне рабочего каталога.

**Права по ролям:**

| Агент | edit | bash | webfetch |
|---|---|---|---|
| `team-lead` | только `work-area/{sessions,tasks,memory}/**` | deny | deny |
| `explorer` | только `explore-report.md` | deny | deny |
| `analyst` | только `requirements.md` | deny | deny |
| `architect` | только `design.md` | deny | deny |
| `coder-front` | зоны фронтенда *(проектный конфиг)* | allow + денилист | deny |
| `coder-back` | зоны бэкенда *(проектный конфиг)* | allow + денилист | deny |
| `tester` | `test-results/**`, `tests/**` — **не исходники** | allow + денилист | deny |
| `guardian` | только `guardian-*.md` | allow + денилист | deny |
| `devops` | инфра-файлы + `devops-status.md` | allow + денилист | deny |
| `reviewer` | только `final-report-*.md` | deny | deny |
| `ai-engineer` | ask | ask | ask |

Денилист `bash` блокирует разрушительные и сетевые команды (`rm -rf`, `chmod 777`, `git push`, `curl`, `wget`, `ssh`, `nc`), закрывая каналы порчи и эксфильтрации. `sudo` запрещён у всех исполнителей, кроме `devops`, где он переведён в `ask` (инфраструктурные операции под подтверждение). `git commit` у кодеров требует подтверждения (`ask`), а у `tester`/`guardian`/`devops` запрещён (`deny`) — фиксировать изменения должны только кодеры и только осознанно. Ключевые принципы заложены сознательно: чистые роли (explorer/analyst/architect/reviewer) не могут ни исполнять, ни править код; аудитор (`guardian`) не может править код (разделение обязанностей); `tester` не может править исходник, чтобы «подогнать» его под зелёные тесты.

См. также [«Известные ограничения»](#известные-ограничения) — там описано, что эта модель **не** закрывает.

## Структура конфигурации

```
global-config/                  → копируется в ~/.config/opencode/
├── opencode.json               # модели, права (process-level), глобальные предохранители
├── AGENTS.md                   # общие правила, модель исполнения навыков, матрица ролей и прав
├── agents/                     # определения 11 агентов
│   ├── team-lead.md
│   ├── explorer.md
│   └── ...
└── skills/                     # 22 скилла + карта и линтер
    ├── README.md               # карта навыков (реестр: кто/когда/порядок/фолбэк)
    ├── check-skill-map.sh      # линтер целостности карты (навык ↔ строка)
    ├── session-init/           # каждый навык — каталог с SKILL.md
    ├── memory-retrieve/
    ├── guardian-verdict/
    └── ...

project-config/                 → копируется в корень вашего проекта
├── opencode.json               # пути edit для кодеров, LSP — ПРОЕКТНО-СПЕЦИФИЧНО
└── AGENTS.md                   # контекст и конвенции конкретного проекта
```

**Рабочий каталог агентов** (`work-area/`) создаётся в корне проекта; агенты пишут туда отчёты, задачи, сессии и память:

```
work-area/
├── docs/        # explore-report.md, requirements.md, design.md
├── tasks/       # задачи делегирования
├── sessions/    # состояние сессий
├── memory/      # MALMAS: procedural / feedback / conceptual + global-memory
└── reports/     # test-results/, guardian-*.md, devops-status.md, final-report-*.md
```

## Требования

- **opencode** актуальной версии. **Важно:** модель прав опирается на glob-шаблоны путей в ключах `edit`/`read` и шаблоны команд в `bash`. Убедитесь, что ваша версия их поддерживает (`opencode --version`); на старых версиях, понимающих только строковые `allow|ask|deny`, привязку к путям придётся упростить.
- **Провайдер моделей.** Роли используют модели `ollama-cloud/*`. Настройте соответствующий провайдер/эндпоинт по [документации провайдеров opencode](https://opencode.ai/docs/providers/) либо замените модели на доступные вам (см. [«Кастомизация моделей»](#кастомизация-моделей)).
- **Node.js** — для LSP-серверов, запускаемых через `npx` (pyright, typescript-language-server, yaml/bash-language-server). Дополнительно `clangd` для C/C++, если он нужен проекту.

## Развёртывание

1. **Установите opencode:**
   ```bash
   npm install -g opencode-ai@latest
   # либо: curl -fsSL https://opencode.ai/install | bash
   opencode --version
   ```

2. **Разверните глобальный слой:**
   ```bash
   mkdir -p ~/.config/opencode
   cp -r global-config/* ~/.config/opencode/
   ```
   После копирования в `~/.config/opencode/` окажутся `opencode.json`, `AGENTS.md`, `agents/`, `skills/`.

3. **Проверьте, что JSON валиден:**
   ```bash
   python3 -c "import json; json.load(open('$HOME/.config/opencode/opencode.json')); print('OK')"
   ```

4. **Настройте провайдер** для выбранных моделей (через `/connect` в TUI или переменные окружения провайдера). Учётные данные opencode хранит в `~/.local/share/opencode/auth.json`.

5. **Подготовьте проект** — см. следующий раздел.

## Адаптация под свой проект

Глобальный слой переносится без изменений. Адаптируется **только проектный слой**. Сделать это можно двумя путями: **автоматически** (агент `ai-engineer` сам сканирует репозиторий и формирует проектный конфиг под вашим контролем) или **вручную**. Описаны оба — ручные шаги полезно прочитать в любом случае: они объясняют, что именно генерирует агент, и помогают проверить его результат.

### Автоматическая адаптация через `ai-engineer` (рекомендуется)

`ai-engineer` — primary-агент вне основного конвейера с правами `edit/bash/webfetch: ask`. Он единственный, кто может *записать* проектный конфиг, и делает это только с вашим подтверждением на каждое изменение — что для файла, отвечающего за права доступа, как раз и нужно. Глобальный денилист секретов при этом действует, поэтому при сканировании он не прочитает `.env` и ключи.

**Вызов** из корня вашего проекта:

```bash
# одноразово (batch-режим):
opencode run --agent ai-engineer "<промпт ниже>"

# либо интерактивно: запустите `opencode`, клавишей Tab переключитесь на ai-engineer и вставьте промпт
```

**Готовый промпт** (вставьте как есть, при желании уточните стек в первой строке):

> Ты адаптируешь конфигурацию мультиагентной системы opencode под текущий проект. Действуй строго в следующих рамках.
>
> **1. Просканируй репозиторий (только чтение).** Определи: язык(и) и стек; каталоги фронтенда и каталоги бэкенда (это будущие зоны записи для `coder-front` и `coder-back` — они НЕ должны пересекаться); где лежат тесты; инфраструктурные файлы (docker-compose, nginx, Makefile, Helm, Terraform и т.п.). Не открывай и не выводи содержимое секретов (`.env`, ключи) — они и так закрыты на чтение.
>
> **2. Сформируй/обнови проектный `opencode.json` в корне проекта.** В нём:
> - `agent.coder-front.permission.edit` и `agent.coder-back.permission.edit` по схеме: первым правилом `"*": "deny"`, затем конкретные каталоги со значением `"allow"`. Перечисли только те зоны, которые роль реально должна менять, без пересечений между фронтом и бэком.
> - Если тесты лежат не в `tests/` — переопредели `agent.tester.permission.edit`, добавив реальный путь тестов; писать в исходники `tester` не должен.
> - Если инфраструктура отличается от `docker-compose.yml`/`nginx.conf`/`Makefile`/`.env*` — переопредели `agent.devops.permission.edit` под реальные файлы.
> - Блок `lsp` — под обнаруженные языки проекта.
>
> **3. Сформируй/дополни `AGENTS.md` в корне проекта:** назначение проекта, структура каталогов, ключевые конвенции кода и специфику, которую агентам нужно учитывать.
>
> **4. Запреты.** Не трогай глобальный конфиг (`~/.config/opencode/`). Не ослабляй и не добавляй глобальные права. Не открывай чтение секретов. Не добавляй `permission.skill`. Не выдавай агентам прав сверх их зон.
>
> **5. Перед записью** покажи каждый файл на подтверждение и кратко обоснуй выбранные зоны записи (почему именно эти каталоги отнесены к фронту/бэку).
>
> **6. После записи** выведи команду для проверки того, что у кодеров корректно смержились права: `edit` из проектного конфига и ограниченный `bash` из глобального.

После генерации `ai-engineer` покажет каждый файл на подтверждение. Примите изменения и выполните [проверку после установки](#проверка-после-установки) — в первую очередь проверку мержа прав кодера.

### Ручная адаптация

Если предпочитаете править руками — скопируйте `project-config/` в корень репозитория и отредактируйте по шагам ниже.

#### 1. Пути исходников для кодеров (`project-config/opencode.json`)

Это обязательный шаг: в глобальном конфиге кодеры имеют `edit: {"*": "deny"}` (безопасный дефолт), а реальные зоны записи открываются здесь, под раскладку вашего репозитория. Пример (Python-проект):

```json
{
  "$schema": "https://opencode.ai/config.json",
  "agent": {
    "coder-front": {
      "permission": { "edit": { "*": "deny", "static/**": "allow", "templates/**": "allow" } }
    },
    "coder-back": {
      "permission": { "edit": { "*": "deny", "src/**": "allow", "core/**": "allow", "services/**": "allow" } }
    }
  }
}
```

Принципы: catch-all `*` всегда первым со значением `deny`, конкретные зоны — следом со значением `allow`. Перечисляйте все каталоги, которые кодер должен иметь право менять, и ничего сверх того. Убедитесь, что зоны фронта и бэка не пересекаются.

#### 2. Каталог тестов

Если тесты лежат не в `tests/`, поправьте allow-путь у `tester` (в глобальном `opencode.json` или переопределите в проектном). `tester` сознательно лишён права писать в исходники.

#### 3. Инфраструктурные файлы (`devops`)

Глобально `devops` может править `docker-compose.yml`, `docker-compose.*.yml`, `nginx.conf`, `Makefile`, `.env*`. Если ваша инфраструктура иная (Helm, Terraform, иные имена файлов) — переопределите его `edit` в проектном конфиге.

#### 4. LSP под языки проекта (`project-config/opencode.json`)

Задайте языковые серверы под стек проекта. Пример для Python с pyright:

```json
"lsp": {
  "python": {
    "command": ["npx", "--yes", "--package=pyright", "--", "pyright-langserver", "--stdio"],
    "extensions": [".py", ".pyi"],
    "initialization": { "settings": { "python": { "pythonPath": "<путь к venv python>" } } }
  }
}
```

#### 5. Контекст проекта (`project-config/AGENTS.md`)

Опишите назначение проекта, структуру каталогов, конвенции кода и специфику, которую агенты должны учитывать. Этот файл — основной источник доменного контекста для всех агентов.

### Автоматическая адаптация через `ai-engineer`

Шаги 1–5 можно не делать вручную, а поручить агенту `ai-engineer`. Это primary-агент, он стоит вне конвейера и предназначен для прямого вызова; его права `edit`/`bash` установлены в `ask`, то есть **каждое изменение он показывает на подтверждение** — что и нужно при правке файла, управляющего безопасностью.

**Перед запуском:** глобальный слой уже должен быть установлен (см. [«Развёртывание»](#развёртывание)), а команда — запускаться **из корня адаптируемого проекта**.

```bash
cd /путь/к/вашему/проекту
opencode --agent ai-engineer
```

Затем вставьте промпт целиком (в TUI можно также переключиться на агента клавишей Tab):

```text
Ты адаптируешь конфигурацию мультиагентной системы под текущий проект.
Действуй строго по шагам и ничего не записывай без моего подтверждения.

1. ИССЛЕДОВАНИЕ (только чтение). Изучи структуру репозитория и определи:
   - каталоги исходников фронтенда;
   - каталоги исходников бэкенда;
   - каталог тестов;
   - инфраструктурные файлы (docker-compose, nginx, Makefile, Helm, Terraform и т.п.);
   - языки проекта и подходящие LSP-серверы.
   Не открывай файлы секретов (.env, *.key, *.pem, secrets/** и подобные).

2. ПРЕДЛОЖЕНИЕ. Покажи таблицей, что собираешься записать:
   - зоны edit для coder-front и coder-back (по обнаруженным каталогам);
   - при необходимости — override для tester (каталог тестов) и devops (инфра-файлы),
     если они отличаются от значений по умолчанию;
   - блоки lsp по обнаруженным языкам.
   Дождись моего подтверждения.

3. ЗАПИСЬ (после подтверждения). Создай/обнови в КОРНЕ проекта:
   - ./opencode.json — проектные права кодеров и lsp;
   - ./AGENTS.md — назначение проекта, структура каталогов, конвенции.

ОГРАНИЧЕНИЯ (соблюдай неукоснительно):
   - в каждом блоке edit первым правилом ставь "*": "deny", конкретные зоны — следом "allow";
   - не расширяй зоны edit за пределы реально обнаруженных каталогов исходников;
   - зоны фронтенда и бэкенда не должны пересекаться;
   - никому не давай право read на секреты;
   - не изменяй глобальный конфиг (~/.config/opencode) и его хардненинг;
   - process-уровневые права (bash-денилисты, права чистых ролей) не трогай — только проектные edit-зоны кодеров, lsp и AGENTS.md.
```

> **Проверьте предложенные пути перед подтверждением.** Агент пишет именно тот файл, который определяет, куда кодеры могут писать. Убедитесь, что зоны не шире необходимого и не пересекаются, прежде чем одобрять запись. После завершения выполните [проверку мержа прав кодера](#проверка-после-установки).

#### Что НЕ трогать

Глобальный хардненинг — переносимая основа: денилист на чтение секретов, `external_directory: deny`, права чистых ролей, bash-денилисты исполнителей. Менять их под проект не нужно и не рекомендуется.

## Проверка после установки

- **Валидность JSON.** Глобальный `opencode.json` — строгий JSON, проверяется командой из шага 3 развёртывания (`python3 -c "import json; ..."`). Проектный `./opencode.json` использует **JSONC** (содержит комментарии `//`), поэтому строгий `json.load`/`json.tool` на нём ожидаемо падает — проверяйте его JSONC-валидатором или запуском opencode.
- **Мерж прав кодера (критично).** Кодер получает `edit` из проектного конфига, а `bash`/`webfetch` — из глобального. Это работает только если opencode мержит права по под-ключам. Запустите кодера на тестовой задаче и убедитесь, что он одновременно (а) может писать в свои зоны и (б) имеет ограниченный `bash` из глобального денилиста. Если `bash` у кодера оказался незаданным или полностью открытым — проектный блок затёр глобальный; продублируйте денилист в проектный конфиг.
- **Целостность карты навыков.** Запустите `sh ~/.config/opencode/skills/check-skill-map.sh` (или из репозитория `sh global-config/skills/check-skill-map.sh`) — он подтвердит, что каждый навык на диске присутствует в карте (`skills/README.md`) и наоборот. `exit 0` — всё согласовано. Полезно после добавления/удаления навыков.
- **Денилист секретов.** Убедитесь, что попытка чтения `.env` агентом отклоняется.
- **Сухой прогон.** Дайте простую задачу и проверьте, что конвейер проходит по этапам, отчёты пишутся в `work-area/reports/`, а решение принимается корректно.

## Эксплуатация

Запуск в каталоге проекта:

```bash
opencode            # интерактивный TUI
# либо одноразово:
opencode run "Опишите задачу здесь"
```

Задача поступает к `team-lead`, который оценивает сложность, ведёт её по конвейеру и возвращает результат. Человеческие точки контроля: выбор варианта на первом цикле и эскалации (исчерпание циклов доработки, плато, инциденты). Решение ACCEPT при успешной верификации может приниматься автономно — если вам нужен обязательный человеческий гейт перед коммитом, добавьте его на уровне процесса (например, запретив автонокоммит; `git push` уже в денилисте).

### Кастомизация моделей

Модели задаются полем `model` у каждого агента в `opencode.json`. Замените `ollama-cloud/*` на доступные вам через `/models` или конфигурацию провайдера. Рекомендация: координатору и аудитору — модели посильнее и с низкой «температурой», исполнителям — быстрые кодовые модели.

## Известные ограничения

- **Обход edit через bash.** Для пяти агентов с `bash` (кодеры, `tester`, `guardian`, `devops`) привязка `edit` к путям обходится записью через шелл (`echo >`, `cat >`). Денилист гасит разрушительные и сетевые команды, но не межзональную запись. Бэкстопом служит ретро-проверка Scope Compliance у `guardian`. Полная герметизация появится, когда opencode выпустит per-agent filesystem boundaries.
- **Доступ `devops` к `.env`.** Если `devops` получает read-override на `.env` (для чтения существующих значений), это единственное место, где сырые секреты попадают в контекст модели. Если достаточно генерировать `.env` из несекретного `.env.example` — оставьте только право `edit`, без `read`.
- **Единая точка координации.** `team-lead` — единственный маршрутизатор и решающий узел; деградация его маршрутизации затрагивает всю систему. Для текущего масштаба приемлемо как осознанный компромисс.
- **Экономика полного конвейера.** Обязательный проход всех этапов на каждом цикле (включая повторное исследование на доработке) — основной источник расхода токенов. При высокой нагрузке имеет смысл ввести условный пропуск исследования на доработке и быстрый путь для тривиальных задач.
