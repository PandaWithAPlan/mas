# Чеклист адаптации development-team под проект

Версия чеклиста: 1.3.0  
Система: OpenCode Multi-Agent System (MALMAS)

Используй этот документ при каждом переносе конфигурации на новый проект.
Все файлы для правки находятся в `project-config/`.
Файлы `global-config/` — не трогать (они портируемые).

---

## Шаг 1 — Скопировать project-config в корень репозитория проекта

opencode читает проектный конфиг из корня проекта (`./opencode.json`, `./AGENTS.md`).
Поэтому копируй **содержимое** `project-config/`, а не саму директорию:

```bash
cp development-team/project-config/opencode.json /path/to/your-project/opencode.json
cp development-team/project-config/AGENTS.md     /path/to/your-project/AGENTS.md
```

Оба файла нужно адаптировать перед запуском агентов.

- [ ] `opencode.json` и `AGENTS.md` лежат в корне проекта (не во вложенной `project-config/`)

## Шаг 2 — Заполнить `opencode.json`

### 2.1 Зоны редактирования coder-front

Найди секцию `agent.coder-front.permission.edit`, раскомментируй строку и замени `FRONTEND_DIR`:

```json
"coder-front": {
  "permission": {
    "edit": {
      "*": "deny",
      "frontend/**": "allow"      // ← замени frontend/ на реальную директорию
    }
  }
}
```

- [ ] Указана реальная директория фронтенда

### 2.2 Зоны редактирования coder-back

Аналогично для `agent.coder-back.permission.edit`. Можно указать несколько директорий:

```json
"coder-back": {
  "permission": {
    "edit": {
      "*": "deny",
      "src/**": "allow",
      "services/**": "allow"
    }
  }
}
```

- [ ] Указаны реальные директории бэкенда (одна или несколько)
- [ ] Зоны `coder-front` и `coder-back` не пересекаются

### 2.3 LSP-серверы

Оставь только языки, которые реально используются в проекте. Удали остальные секции.

- [ ] **Python** — если используется: заменить `/path/to/venv/bin/python` на реальный путь к venv.
  Если venv не нужен — удали блок `initialization` целиком.
- [ ] **TypeScript / JavaScript** — оставить если есть `.ts`/`.js` файлы, иначе удалить
- [ ] **C / C++** — заменить `/path/to/clangd` на реальный путь (`which clangd`), или удалить
- [ ] **YAML** — оставить если есть Docker Compose / CI конфиги, иначе удалить
- [ ] **Shell** — оставить если есть `.sh` скрипты, иначе удалить

## Шаг 3 — Заполнить `AGENTS.md`

Шаблон содержит пошаговую инструкцию «КАК АДАПТИРОВАТЬ ЭТОТ ШАБЛОН» и блоки с пометкой
`[ЗАПОЛНИТЬ]` (внутри HTML-комментариев `<!-- ... -->`). Заполни их по порядку.

### 3.1 Назначение директорий

Секция **«Назначение директорий»** (блок `[ЗАПОЛНИТЬ]` с деревом каталогов).
Замени дерево с плейсхолдерами на реальную структуру проекта и укажи владельца каждой директории.

- [ ] Указан реальный корень проекта и точка входа (main, index, app и т.д.)
- [ ] Указаны директории фронтенда (владелец `coder-front`)
- [ ] Указаны директории бэкенда (владелец `coder-back`)
- [ ] Указана директория тестов
- [ ] Поддерево `work-area/` оставлено без изменений (оно стандартное)

### 3.2 Технологический стек

Секция **«Технологический стек и ключевые подсистемы»** (таблица с пометкой `[ЗАПОЛНИТЬ]`).

- [ ] Указан язык и версия (Python 3.11, Node 20, Go 1.22 и т.д.)
- [ ] Указан фреймворк (FastAPI, Express, Gin и т.д.)
- [ ] Указана СУБД (PostgreSQL, MongoDB, SQLite и т.д.)
- [ ] Указана контейнеризация / тестовый фреймворк
- [ ] Удалены неиспользуемые строки таблицы

### 3.3 Ключевые внешние зависимости

Подсекция **«Ключевые внешние зависимости»** (блок `[ЗАПОЛНИТЬ]`).

- [ ] Перечислены все внешние сервисы (базы данных, кэши, очереди, мониторинг)
- [ ] Перечислены MCP-серверы, если используются (и их эвристическое имя для `agent.devops`)
- [ ] Указаны известные проблемы / связанные эвристики; если зависимостей нет — подсекция удалена

### 3.4 Зоны ответственности агентов

Секция **«Зоны ответственности»** (блок `[ЗАПОЛНИТЬ]` со строками `[FRONTEND_DIR]` / `[BACKEND_DIR_*]`).

- [ ] `coder-front` привязан к реальным директориям фронтенда
- [ ] `coder-back` привязан к реальным директориям бэкенда
- [ ] Значения **совпадают** с `opencode.json → agent.coder-*.permission.edit`

### 3.5 Локальные уточнения процесса

Секция **«Применение глобального процесса»** (блок `[ЗАПОЛНИТЬ]` в конце файла).

- [ ] Кратко перечислены ключевые локальные уточнения (зоны, нумерация, DevOps Gate и т.д.)
- [ ] Добавлены project-специфичные ограничения, если есть
  - Примеры: "не изменять файлы миграций вручную", "CSS только через Tailwind",
    "все API endpoints через OpenAPI схему"
- [ ] Удалены все оставшиеся блоки `[ЗАПОЛНИТЬ]`

## Шаг 4 — Создать work-area/ в проекте

Пути ниже соответствуют тем, что прописаны в `global-config/opencode.json`
(в частности, отчёты Tester лежат в `work-area/reports/test-results/`).

```bash
mkdir -p work-area/memory
mkdir -p work-area/docs
mkdir -p work-area/tasks
mkdir -p work-area/sessions
mkdir -p work-area/reports/test-results
touch work-area/memory/procedural.json
touch work-area/memory/feedback.json
touch work-area/memory/metrics.json
touch work-area/memory/conceptual.md
touch work-area/memory/global-memory.md
```

Инициализируй `procedural.json`, `feedback.json` и `metrics.json` как пустые JSON-массивы:

```json
[]
```

- [ ] Директория `work-area/` создана со всеми поддиректориями (`memory/`, `docs/`, `tasks/`, `sessions/`, `reports/test-results/`)
- [ ] `procedural.json`, `feedback.json`, `metrics.json` инициализированы как `[]`
- [ ] `conceptual.md` и `global-memory.md` созданы (могут быть пустыми)

## Шаг 5 — Проверить конфигурацию перед первым запуском

- [ ] Нет оставшихся плейсхолдеров `[ЗАПОЛНИТЬ]`, `/path/to/`, `FRONTEND_DIR`, `BACKEND_DIR`
- [ ] `opencode.json` валиден. **Важно:** проектный `opencode.json` содержит комментарии `//`
  (формат JSONC), поэтому `python -m json.tool` / `json.load` на нём **упадёт** — это нормально.
  Проверяй его JSONC-валидатором (например, `npx jsonc-parser`) или просто запуском opencode.
  Строгая проверка `python -m json.tool` подходит только для `global-config/opencode.json` (без комментариев).
- [ ] Зоны `coder-front` и `coder-back` в `opencode.json` совпадают с тем, что написано в `AGENTS.md`
- [ ] Удалены LSP-серверы для неиспользуемых языков
- [ ] **Мерж прав кодера проверен** (критично): кодер одновременно может писать в свои зоны
  (`edit` из проектного конфига) и имеет ограниченный `bash` из глобального деналиста.
  Если `bash` у кодера открыт настежь — продублируй bash-деналист в проектный конфиг
  (см. README, раздел «Проверка после установки»).

## Шаг 6 — Первый запуск

1. Запустить `session-init` (team-lead создаст `work-area/sessions/session-YYYY-MM-DD.md`)
2. Дать первую задачу team-lead
3. Убедиться, что `memory-retrieve` не падает на пустых файлах памяти
